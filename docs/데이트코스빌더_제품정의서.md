# 데이트 코스 빌더 — 제품 정의서

> **핵심 철학 (이 앱의 전부):**
> **AI가 판단하지 않는다. 정확한 데이터로 후보를 많이 펼치고, 유저가 직접 보고 고르고 조립한다.**
>
> 구글 Places의 실제 가격대·영업시간·평점을 그대로 보여주고 유저가 직접 판단한다. 앱이 판단을 안 하니까 틀릴 여지가 없다. 데이터만 정확히 나르면 된다.

---

## 한 줄 정의

도쿄 현지인이 하루 데이트 코스를 **직접 골라 조립**하는 앱. AI 추천 없이, 정확한 데이터로 후보를 펼치고 유저가 선택한다.

---

## 핵심 은유 — "컨베이어벨트 조립"

이 앱의 정체성을 한 단어로: **컨베이어벨트.**

**지금 유저가 겪는 노동 (이 앱이 없앨 것):**
데이트 하나 짜려면 구글에서 따로따로 검색한다 —
1. "우에노 데이트 장소" 검색 → 탭
2. "그다음 어디 가지" → "도쿄 데이트 장소" 검색 → 딴 탭
3. "우에노 맛집" 검색 → 딴 탭
4. "우에노 야경" 검색 → 딴 탭
5. 이걸 다 머릿속/메모장으로 수동 조립

= **검색 5번 + 탭 여러 개 + 수동 조립.** 매번 반복되는 노동.

**이 앱이 하는 것:**
그 흩어진 검색·조립을 **하나의 흐름(레일)으로 이어붙인다.** 지역 고르면 후보 뜨고 → 고르면 다음 단계(맛집)로 자동으로 넘어가고 → 고르면 또 다음(야경)으로 → 끝에 코스 완성. 유저가 손으로 하던 "검색→탭→검색→탭→조립"을 컨베이어벨트 하나로.

**왜 이게 구글맵과 근본적으로 다른가:**
- **구글맵 = 점 하나를 찾는 도구** (검색). 5번 따로 해야 함.
- **이 앱 = 점들을 선으로 잇는 도구** (조립). 컨베이어벨트로 한 번에.
- 구글맵의 근본 구조는 "검색"이지 "순차 조립"이 아니다. "데이트 모드" 하나 추가한다고 되는 게 아니라 **구조 자체가 다르다.**

**여행 빌더와도 다른 점:**
- 여행 빌더 = 빈 캔버스에 알아서 하나씩 검색해 채우기
- 이 앱 = **레일 따라 순서대로 안내** (지역→장소→맛집→야경→완성)
- 데이트처럼 "뭐부터 정해야 할지 막막한" 상황엔 레일이 훨씬 편하다.

→ **이 "컨베이어벨트" 구조가 곧 UI 흐름의 핵심**이다. 한 화면에 다 담지 않고, 단계가 순서대로 흐르게 설계한다.

---

## 문제 (왜 만드는가)

데이트 장소를 정하는 게 **매번 반복되는 고민**이다. 두 가지 진짜 원인:

1. **아이디어 고갈** — 창의력 부족이 아니라 **아는 장소가 한정**돼서다. 후보를 많이 보면 고를 수 있다.
2. **AI 추천의 부정확함** — 챗지피티·기존 AI 플래너에 물어보면 **가격이 틀리고, 매진된 곳을 추천하고, 예약 정보가 틀린다.** 결국 손수 다 찾게 된다.

→ **해법:** 앱이 판단·추천하지 않는다. 정확한 데이터(구글 Places)로 후보를 왕창 펼치고, 유저가 직접 보고 판단한다.

## 타겟

**데이트 장소를 매번 고민하는 사람.** (제작자 본인이 그 증거 = 최소 사용자 1명 확보 + 니즈 100% 이해)

- 여행자용 앱과 달리 **현지인** 기준: "관광지"가 아니라 "우리 동네에서 갈 만한 곳"
- 반복 사용: 데이트는 계속 하니까 1회성이 아님
- **지역 범위 = 전 세계 (구글 Places가 국경 무관)**: 앱은 국내/해외를 구분 안 함 — 화면 1.5에서 받는 "지역 이름"만 구글에 넘김. 오키나와·방콕·파리 어디든 동일하게 작동. (오키나와·태국으로 검증 완료)
- **정체성은 여전히 "데이트 코스 빌더" (여행 앱 아님)**: 전 세계 작동은 "구글 데이터라 공짜로 따라오는 보너스"지, 정체성을 "여행 앱"으로 바꾸는 게 아님. 여행 앱(Wanderlog 등)으로 정체성을 틀면 차별점(일상 데이트 특화)이 흐려짐. 뿌리는 데이트 코스, 범위만 전 세계.
- **사용 상황**: 일상 데이트(간토 등) + 여행지 데이트(국내외). 여행은 낯선 곳이라 "뭐 하지" 고민이 더 커서 오히려 니즈 강함. → 앱 열 이유가 많아져 "사용 빈도 낮음" 약점 완화.
- **검증**: 주 사용(제작자의 일상 데이트, 간토)으로 먼저 검증 → 여행지는 구조상 자동으로 따라옴. 확장 시 갈아엎을 코드 없음 (지역 하드코딩 안 함).
- **기본 지역 목록 = 제한이 아니라 편의**: 자주 가는 곳을 바로가기로. 안 적은 곳(국내외 불문)도 검색·카테고리 입구로 다 나옴. (오키나와에서 지역목록 없어도 카테고리로 다 발견됨을 검증)

## 차별점 (기존 앱을 안 쓰고 이걸 쓸 이유)

경쟁 지형을 다 확인한 결과, 두 종류의 기존 앱과 명확히 구별된다.

**1. 여행 일정 빌더 (Wanderlog, Map Your Voyage, Travaa 등)와 다른 점**
- 그들: 며칠짜리, 다도시, 항공·호텔 관리, 관광객용 → 무겁고 복잡
- 이 앱: **하루, 데이트, 현지** 특화 → 가볍고 딱 맞음

**2. AI 데이트 앱 (DateMore, Cupla, SoulPlan 등)과 다른 점**
- 그들: AI가 취향 분석해서 추천, 가끔 정보 틀림
- 이 앱: **AI 판단 없음. 데이터 펼치고 유저가 판단** → 틀릴 여지 없음

**한 문장:** 여행앱이 아니라 데이트앱이고, AI 추천앱이 아니라 데이터+유저판단 앱이다.

---

## 핵심 흐름 (유저가 앱에서 하는 것)

이 대화에서 실제로 돌려본 그 흐름 그대로:

1. **시간 정하기** — 그날 데이트를 몇 시부터 몇 시까지 할지 (예: 12시~23시)
2. **지역별 후보 펼치기** — 여러 지역을 넘나들며, 카테고리 필터(볼거리·체험·카페 등)로 후보를 **왕창** 보여줌
3. **유저가 선택** — 후보를 보고 갈 곳을 직접 고름 (앱은 추천 안 함, 나열만)
4. **음식점 후보 펼치기** — 고른 곳들 동선에 맞춰 음식점을 또 왕창 → 유저 선택
5. **자동 조립** — 1번의 시간에 맞춰, 고른 것들을 **동선·시간표(코스)로 조립**해서 보여줌

**중요 원칙:** 유저에게 취향을 인터뷰하지 않는다("분위기 어때?" 같은 질문 X). **카테고리 필터만 주고 후보를 펼친다.** 선택은 유저가.

## 데이터

- **구글 Places API** — 장소·음식점의 이름, 위치, 영업시간, 평점, 가격대
- 이 데이터를 **해석하지 않고 그대로 보여준다** (AI가 "이 집 3천엔" 하고 틀리는 대신, 실제 가격대·영업시간을 그대로)
- 데이터 벽 없음: 이 대화에서 실제로 구글 Places로 후보 뽑고 코스 짜는 걸 증명함

**데이터 방식: A (순수 데이터, 큐레이션 없음)**
- 앱이 "이게 좋은 데이트 장소야"라고 **판단·큐레이션하지 않는다.** 구글 Places 결과를 그대로 펼친다.
- 대신 **필터를 유저 손에 쥐여준다** — 이건 큐레이션이 아니라 도구 제공:
  - "평점 4.0 이상만"
  - "지금 영업중만"
  - "가격대 ~3천엔"
  - "카페 / 볼거리 / 체험" 카테고리 탭
- 앱은 판단 안 하고 필터만 줌 → 걸러진 결과는 **유저가 스스로 만든 것.** 철학("유저가 판단") 유지.
- **A의 트레이드오프 (알고 감):** 데이터가 구글맵과 같으므로, 차별점은 ①컨베이어벨트 조립 ②데이트 특화 필터, 이 둘에 걸린다. 조립 경험이 매끄럽지 않으면 유저는 구글맵으로 돌아간다. 그러니 조립 UX가 생명.

## 버전1(MVP) vs 버전2 — 한눈 비교

| 기능 | 버전1 (MVP) | 버전2 (나중) |
|---|---|---|
| **코스 짜기 (화면 0~4)** | ✅ 전부 | — |
| **세 입구 검색** (지역·카테고리·검색) | ✅ | — |
| **여친 정보 참고 표시** | ✅ (보기만) | 자동 필터링 |
| **마이 노트** (기념일·할일·여친정보) | ✅ 내 폰에만 | 여친과 공유 |
| **카드 메모** | ✅ | — |
| **편집 타임라인** (드래그·삭제) | ✅ | — |
| **구글맵 경로 보기** | ✅ 구글맵 앱으로 링크 | 앱 내 지도 |
| **호텔** | ✅ 화면·위치·평점 (화면5) | 실시간 가격·빈방·**예약** |
| **코스 저장·기록** | ✅ localStorage | 백엔드(계정·다기기) |
| **공유** | ✅ 이미지/텍스트로 내보내기 | 링크로 앱에서 같이 열기·수정 |
| **사진** | ❌ | ✅ 다녀온 코스에 사진 첨부 (앨범 정리 불필요) |
| **AI 추천** | ❌ | ✅ 가벼운 힌트 수준 |
| **이벤트** (불꽃놀이·마츠리) | ❌ | ✅ 큐레이션, 월 1회 업데이트 |
| **액티비티** (방탈출·클래스) | ❌ | ✅ |
| **리뷰 함정 필터링** (바가지·휴무 경고) | ❌ | ✅ (AI 필요) |
| **다국어** | ❌ (일본어) | ✅ 영어 → 한중 |
| **백엔드** | ❌ 없음 (저장 로직은 추상화) | ✅ 있음 |

**한 줄 요약:**
- **버전1 = "혼자, 코스 짜고 저장하는" 완결된 앱.** 백엔드 없이 돌아감. 핵심 가설(컨베이어벨트가 편한가)을 검증.
- **버전2 = "백엔드 붙여서 추억·공유·예약"까지.** 사진·무거운 공유·호텔 예약·이벤트·AI가 여기서 추가.
- **경계선 = 백엔드.** 백엔드 없이 되는 건 버전1, 백엔드가 필요한 건 버전2. (단 저장 로직을 추상화해둬서 전환 시 갈아엎기 없음)

## MVP 범위 (지금 만들 것)

**넣는다:**
- 시간 정하기
- 지역별·카테고리별 후보 펼치기 (장소 + 음식점)
- 유저 선택
- 동선·시간표 자동 조립
- 데이터: 구글 Places (일본어 기준)

**뺀다 (버전2로):**
- ❌ AI 추천
- ⚠️ 호텔 — **화면·위치·평점은 MVP(화면5)**, 실시간 가격·빈방·예약만 버전2
- ❌ 이벤트 (불꽃놀이, 마츠리)
- ❌ 액티비티 (특수 체험)
- ❌ 리뷰 함정 필터링 (바가지·휴무 자동 경고)
- ❌ 다국어 (영어 등 — 제작자가 일본어로 테스트 가능하므로 나중에)

**뺀 이유:** 핵심 가설("데이터 펼침 + 유저 조립"이 실제로 쓸 만한가)을 최소 노력으로 확인하는 게 MVP 목적. 위 기능들은 그게 검증된 뒤 얹는다.

## 버전2 이후 (나중에)

MVP는 순수 데이터(A)지만, 버전2 기능들은 **데이터 성격이 제각각**이다. 이 구분이 중요:

- **이벤트 (불꽃놀이·마츠리)** — **큐레이션 방식.** 실시간 API가 없으므로 직접 큐레이션해서 넣고, **한 달에 한 번 업데이트.** MVP의 순수 데이터와 성격이 다름(이건 사람이 고른 목록). "이번 달 데이트하기 좋은 이벤트" 느낌.
- **AI 추천** — 판단이 아니라 "이런 조합 어때?" 수준의 가벼운 힌트로. MVP 철학(AI 판단 없음)과 충돌하지 않는 선에서.
- **호텔 예약** — 호텔 화면·위치·평점은 이미 MVP(화면5). 버전2는 **실시간 가격·빈방·예약 연동**만. affiliate API(라쿠텐트래블·부킹닷컴 등).
- **액티비티** — 방탈출·원데이클래스 등 특수 체험 (구글 Places로 일부 잡히나 정리 필요)
- **리뷰 함정 필터링** — 리뷰 읽고 바가지·휴무·외국인거부 경고 (무거움, AI 필요)
- **다국어** — 영어부터 → 한국어·중국어
- **알림** — 기념일 알림, 예정 데이트 알림 (웹 푸시: Service Worker·권한 처리 필요 — V1엔 무거움)
- **다크 테마** — 전 화면 CSS 두 벌 관리
- **데이터 백업·복원** — V2 백엔드(서버) 생기면 자연스럽게 (localStorage만으론 반쪽)
- **이미지 공유** — 코스를 예쁜 이미지 카드로 만들어 공유 (V1은 텍스트+구글맵 링크)
- **직접 예약 (앱 안 완결)** — V1은 공식 사이트로 연결까지
- ※ 위 항목들은 V1 내정보 탭·공유·상세 화면 확정 때 "좋지만 V1 범위 밖"으로 분류된 것들 (2026-07 스케치 FINAL 점검에서)

**핵심 구분:**
- MVP(장소·맛집) = **순수 데이터, 큐레이션 없음** (구글 Places 실시간)
- 버전2 이벤트 = **큐레이션 있음, 월 1회 수동 업데이트** (실시간 데이터 없어서)
- 두 개는 데이터 소스도 갱신 주기도 다르므로 설계 시 분리해서 다룬다.

## 버전 단계 (무거운 순서대로 나눔)

형이 원하는 것(혼자 조립 → 공유 → 커플 앱)을 **한 번에 안 만들고 단계로 쌓는다.** 커플 공유·채팅을 붙이면 순수 코스 빌더(Wanderlog류)가 아니라 커플 앱(Cupla류)과 경쟁하게 되고, 기술도 몇 배 무거워지므로 신중하게 단계를 나눔.

**MVP — 혼자 쓰는 코스 빌더 (백엔드 없음)**
- 마이페이지(홈) + 컨베이어벨트 조립(화면1~4) + 카드 메모 + 이미지/텍스트 공유
- 마이페이지: 기념일 메모, 예정 코스, 다녀온 코스 기록, 즐겨찾기 — 전부 localStorage
- React + TypeScript + Google Places/Maps API + Vercel
- **저장 로직은 추상화** (saveCourse/loadCourse 한 군데로) → 나중에 백엔드 전환 쉽게. localStorage로 시작하되 갈아엎기 없이 전환 가능하게 설계.

**버전2 — 백엔드 + 추억·공유 (핵심 부가기능)**
- **사진 첨부**: 다녀온 코스에 그날 사진 넣기. "데이트 코스에 사진 자동으로 묶임 → 앨범 뒤질 필요 없이 '우에노 데이트' 열면 그날 사진이 다 있음." (사진 찾기 귀찮음 해결 = 킬러 기능 후보. 사진은 용량 커서 백엔드/저장소 필요)
  - **사진 메모**: 사진 밑에 한 줄 코멘트. (V2 1차)
  - **커버 사진 지정**: 코스 대표 1장 → "다녀온 코스" 목록이 사진 썸네일로 보임. (V2 1차)
  - **각자 하트**: 여친과 각자 사진에 ❤️ 반응. 메모보다 가볍게. (V2 1차)
  - **그의 메모 / 그녀의 메모 나란히**: 같은 사진에 누가 썼는지 구분해서 나란히. (V2 1차, 메모 확장)
- **무거운 공유**: 여친도 앱에서 열어 같이 보고 수정. 계정·커플 연결.
- **여친과 사진 보며 통화** 등 커플 기능
- 기념일도 공유 (우리 기념일)
- 이벤트 큐레이션(월 1회), AI 추천, 호텔도 이 단계
- **채팅은 안 만든다. 카톡/라인에 맡긴다.** (이미 쓰는 걸 또 만들 필요 없음 / 대화 욕구는 위 "사진 메모"로 대체)
- **V2 후반·V3 아이디어 (지금은 안 함)**: 추억 지도(우리가 밟은 곳 핀) / "1년 전 오늘" 회고 / 사진 EXIF 시간순 자동 정렬.

**버전3 — 풀 커플 앱 + 커뮤니티**
- 실시간 동기화, 앱 내 채팅 — **정말 필요하면.** (장소별 댓글 같은 구체적 니즈 있을 때만)
- **유저 코스 공개·구경·가져오기 (커뮤니티):** 유저가 다녀온 코스를 (원하면) 모두에게 공개 → 다른 사람이 구경하고 "내 코스로 가져오기". 
  - **왜 강력한가:** 친구 피드백("계획 기능만으론 약하다, 정보가 필요")의 진짜 답. 형이 정보를 직접 만드는 큐레이션(무거움) 대신, **유저들의 코스가 서로에게 정보·영감이 됨.** "도쿄 데이트 뭐 하지?" → 남들 코스 구경 → 가져오기. 제작자의 원래 문제(아는 게 한정돼 아이디어 고갈)를 유저 코스가 해결.
  - **왜 버전3인가:** 유저가 많아야 코스가 쌓이고, 그래야 이 기능이 빛남 (텅 비면 의미 없음). 그리고 백엔드+계정+공개/비공개 설정+커뮤니티 관리(신고 등)까지 필요해 커플 공유(버전2)보다 무거움. → 유저가 쌓인 뒤.
  - 참고: 트리플·트립닷컴도 "남의 일정 보고 가져오기"가 인기 기능. 단 그들은 여행용, 이건 데이트용.

**핵심 우선순위 (제작자 확인):** 코스 짜기 = 핵심. 추억·공유 = 중요하지만 그 다음. 그래서 코스 짜기부터 검증하고 추억·공유를 얹는 순서. "처음부터 백엔드"는 안 함 — 핵심 가설(컨베이어벨트가 편한가)을 검증하기 전에 몇 달을 백엔드에 쏟는 리스크. 대신 저장 로직 추상화로 나중 전환 비용을 없앰.

**친구 피드백 기록 (실사용자 첫 반응):** "트리플 연애 버전이네. 좋은데 계획 짜는 기능은 사람들이 많이는 안 쓴다. 나중엔 트리플처럼 정보 알려주는 것도 있으면 좋겠다."
- **새겨들을 것:** 계획 기능만으론 약함 → "계획 외에 앱 열 이유(정보)"가 필요. (이미 마이노트·기념일·추억으로 일부 대응 중, 그 방향 강화)
- **조심할 것:** "트리플처럼"을 문자 그대로 따라가면 안 됨. 트리플 정보(날씨·환율·번역·235개 도시 가이드)는 ①여행자용이라 현지 데이트엔 불필요 ②큰 회사(대한항공 자회사)가 수백 명이 만든 규모 → 따라가면 짐. 형 앱식 정보 = 데이트 특화(이벤트·핫플·커플혜택), 또는 위 "유저 코스 공개"로 유저가 정보를 만들게.

**참고 앱: 트리플 (여행 앱, 대한항공 자회사)**
- **참고할 것 (교본으로)**: 동선 보며 일정 짜기 UX, 코스 공개·구경(여행기), 커플/동행 실시간 공유, 정보 곁들이기 방식, 드래그 일정 편집. 1500만 다운로드로 "동선짜기+공유+정보" 조합이 먹힌다는 걸 이미 검증 → 형은 백지가 아니라 검증된 답지를 참고. 막히면 "트리플은 이걸 어떻게 했지?" 열어보기. 리뷰의 불만(드래그 튕김 등)은 미리 피하기.
- **참고 ≠ 복사**: 트리플에서 **데이트에 맞는 것만** 골라 참고. 여행자용 기능(항공·호텔 예약, 235개 도시 가이드, 환율·시차·번역)은 배제 — 현지 데이트엔 불필요 + 큰 회사 영역.
- **트리플과 겹쳐도 다른 물건**: 트리플="여행자 여행 준비 올인원", 이 앱="현지 커플 하루 데이트". 기능이 닮아도 시장·정체성이 다름. 형의 진짜 해자 = **커플 관계 중심**(여친 정보·기념일·추억 아카이브·반복되는 관계) — 트리플이 "데이트 모드" 넣어도 여행 앱 구조론 못 따라오는 곳.

**먼 미래 옵션 — 여행 모드 (버전3 이후):**
- 하루 코스를 **여러 날로 확장** = 여행 일정. 구조상 이미 가능(오키나와·태국 검증). "코스를 날짜별로 여러 개 묶기".
- **단, 데이트 정체성 유지**: "여행 앱"이 되는 게 아니라 "데이트 앱인데 여행도 되는" 선. 트리플 정면충돌(항공·호텔 예약) 영역은 안 건드림.
- 아주 나중에. 지금 할 것과 명확히 구분.

## 기술 스택

**MVP:**
- **프론트**: React + TypeScript (컨베이어벨트 = 상태 흐름으로 자연스럽게 표현)
- **데이터**: Google Places API (장소·맛집)
- **지도**: Google Maps JavaScript API (동선 표시)
- **저장**: localStorage (즐겨찾기·코스, 백엔드 없이)
- **배포**: Vercel or Netlify (무료, GitHub 연결 자동 배포)
- **백엔드**: 없음 (프론트에서 API 직접 호출)

**왜 이 스택:**
- 웹앱부터 → 앱스토어 등록·심사·양대 플랫폼 벽 없이 "만들고 폰에서 바로 확인". 초보에게 이 빠른 사이클이 제일 중요.
- React+TS → 형이 이미 TypeScript 조금 앎. 이 앱이 딱 React가 잘하는 종류(상태에 따라 화면 바뀜). 자료 많음.
- 나중에 반응형으로 폰에서도 앱처럼. 필요하면 그때 네이티브/PWA 검토.

**버전2~3에서 추가:**
- 백엔드 (링크 공유 → 계정·DB·실시간). 이벤트 큐레이션 데이터 관리엔 Python 쓸 수 있음.

**주의 — 비용:**
- Google Places/Maps API는 무료 할당량 있고, 넘으면 과금. 혼자 테스트는 무료 안. 사용자 늘면 "API 호출 캐싱"으로 아껴야 함(MVP 단계선 신경 X).

## 화면 설계 (컨베이어벨트를 화면으로)

컨베이어벨트 = 화면이 순서대로 흐름. 한 화면에 다 안 담고 단계별로.
**단, 시작점은 마이페이지(홈). 컨베이어벨트는 홈에서 출발해 홈으로 돌아오는 순환.**

### 화면 0 — 마이페이지 (홈, 앱의 시작점)
```
┌─────────────────────────────┐
│  📌 마이 노트                 │
│  🎂 기념일                    │
│    여친 생일  7/20 (D-16) 🔁  │ ← 날짜O, D-day 자동. 반복 여부 유저 선택
│    우리 100일 8/3 (D-30)     │
│  ✅ 할 일                     │
│    ☐ 생일 선물 예약           │ ← 체크박스, 완료하면 지움
│    ☐ 목도리 사기             │
│  💝 여친 정보                  │
│    • 갑각류 알레르기          │ ← 날짜X, 상시 참고 (완료 없음)
│    • 조용한 카페 좋아함        │
├─────────────────────────────┤
│  📅 예정된 코스               │
│  [+ 새 코스 만들기]           │ ← 화면 1로
├─────────────────────────────┤
│  [ 다녀온 코스 | 즐겨찾기 ]    │ ← 토글 버튼
│  (토글 상태에 따라 목록 바뀜)  │
└─────────────────────────────┘
```
- **마이 노트 3종 (한 화면에 위아래로, 성격 다름):**
  - **기념일**: 날짜O, D-day 자동 계산, **매년 반복 여부를 유저가 체크박스로 선택** (생일=반복, 100일=1회성)
  - **할 일**: 체크박스, 완료하면 지움 (todo). 날짜 붙일 수 있음
  - **여친 정보**: 완료 없이 계속 쌓이는 참고 목록 (알레르기·취향). → **화면 2·3에서 참고용으로 띄움** (가벼운 방식, 자동 필터링 아님. 유저가 보며 판단 = 철학 일치)
- **다녀온 코스 ↔ 즐겨찾기 = 토글** (둘 다 과거 목록이라 성격 비슷, 공간 절약)
- **왜 홈으로 두나 (전략적)**: 앱 열면 기념일·예정코스·지난기록이 보임 → **데이트 안 하는 날에도 앱 열 이유.** "코스 짜기"만 있으면 가끔 쓰고 잊지만, 기념일·기록이 있으면 계속 열어봄. → 형 앱의 최대 약점(데이트 주 1~2회라 사용 빈도 낮음)을 해결.
- MVP는 전부 localStorage. 기념일 공유·사진은 버전2(백엔드).

**앱 전체 흐름:**
```
화면 0 (홈)
   ├─ [새 코스] → 화면1(시간) → 화면1.5(도/현) → 화면2(장소) → 화면3(맛집) → 화면4(완성)
   │                                                    화면4에서 [호텔 추가] → 화면5(호텔, 선택)
   │                                                    완성하면 이름 붙여 홈에 기록
   ├─ 예정 코스 클릭 → 코스 상세 화면 (화면4 재현, 보기 모드)
   ├─ 다녀온 코스 클릭 → 코스 상세 (카드 누르면 메모·(버전2)사진)
   └─ 즐겨찾기 보기
```
- **화면 4 = "코스 상세 화면"으로 재활용**: 새로 짤 때는 편집 모드로 도착, 저장된 코스 열 때는 보기 모드로 도착. **같은 화면, 모드만 다름** (화면 재활용 = 만들 것 줄고 일관성 생김).
- **편집 모드에서 가능한 것 (전부):** 장소 추가 / 장소 삭제(안 가기로 한 곳 빼기) / 시간대 변경 / 날짜 변경(연기 등) / 카드 순서 변경 / 코스 자체 삭제(아예 안 가게 됨). 즉 Course의 모든 필드(items 추가·삭제·순서, CourseItem.startTime, Course.date, courses에서 제거)를 조작 가능. 보기 모드 = 읽기 전용. ✏️(보기 모드 우상단) → 편집 모드 → [저장하기] → 보기 모드 복귀.
- **홈 마이 노트에 [+] 추가 버튼:** 기념일·할 일·여친 정보 각 영역에 [+] → 입력 팝업 → 목록에 추가. 각 항목에 ✏️🗑️(공용 EditDeleteButtons 부품)로 제자리 수정·삭제. 할 일은 ☐ 탭으로 done 토글.
- **뒤로가기(<) 정책:** 코스 만들기 중에는 **한 단계씩 후퇴** (화면3→2→1.5→1, 담은 장소는 유지 — "지역 다시 고르기" 가능). 화면 1에서 한 번 더 뒤로 = 홈으로 나가는 것이므로 그때만 **"저장 안 하고 나갈까요?" 확인 팝업** (실수로 담은 것 전부 날리는 사고 방지).

### 화면 1 — 시작 (날짜·시간)
- **날짜**: 달력에서 톡 찍기
- **시간**: 시작·끝 둘 다, **알람식 휠**로 (예: 시작 12:00 / 끝 23:00)
- 지역은 여기서 안 함 → 다음 화면
- **이 입력이 뒤 화면 전체의 필터**: 그 시간에 영업하는 곳만 뜨고, 그 시간 안에 코스가 배치됨. 그래서 첫 화면.

### 화면 1.5 — 도/현 선택 (기준점 설정)
- 이번 데이트의 **도/현을 먼저 정함** (도쿄·가나가와·사이타마·군마·이바라키·지바...)
- **왜 필요한가 (기술적):** 구글 Places는 "어디"를 알아야 검색함. 현을 정하면 그게 **검색 기준점**이 되어, 이후 모든 후보가 그 현 기준으로 로드됨.
- **왜 필요한가 (UX):** 형 반경이 간토 전역이라 지역이 수십 개. 현을 먼저 좁히면 화면 2 지역 탭에 **그 현 지역만** 떠서 깔끔 (드릴다운).
- **80%가 한 현 안 데이트** → 현 한 번 고르면 그 안에서 짜기. 나머지 20%(현 넘나듦)는 "직접 추가"로 처리. 대부분 케이스를 단순하게, 예외는 기존 기능으로.

### 화면 2 — 지역·장소 선택 (앱의 심장)
```
💝 여친 정보 (접기/펼치기): 갑각류 알레르기 · 조용한 곳 선호
──────────────────────────
🔍 [ 장소·지역 검색 ]              ← 뭐든 검색 (쇼와기념공원/타치카와/공원)
──────────────────────────
📍 지역: [우에노][시부야][신주쿠][+더보기]  ← 입구1: 지역 바로가기
🎯 종류: [공원][카페][볼거리][체험][쇼핑]   ← 입구2: 카테고리
가격: 저렴 ──●──── 비쌈
──────────────────────────
후보 목록 (선택에 따라 바뀜, 현 안 평점순)
  쇼와기념공원 ⭐4.3 [+담기]
   ...
──────────────────────────
[+ 직접 추가]  ← 구글에 없는 곳 / 다른 현 장소
   선택한 곳 3개 → [다음: 맛집]
```

**핵심 구조 = "입구는 셋, 결과는 하나" (검색창이 메인)**
지역·카테고리·검색 어느 걸로 들어오든 **같은 후보 목록**으로 모임. 입구만 여럿, 도착점은 하나 → 안 헷갈림.
**검색창을 위에 크게(메인), 지역 탭·카테고리는 아래 보조.** 사람들은 검색에 익숙(구글·인스타)하므로 "뭐 찾고 싶으면 위에 친다"가 직관적.

1. **통째 검색 (메인 입구, 타겟형 A)**: 검색창에 **"이케부쿠로 넷카페"처럼 지역+장소를 한 번에**. 목적 명확할 때 제일 빠름. 지역 탭 안 거쳐도 됨.
2. **지역으로 좁히기 (보조, 영감형 B)**: [우에노] → 우에노 근처 후보. 한 지역에서 **여러 곳** 볼 때 좋음(지역 고정하고 카테고리만 바꿈). 지역 탭에 없으면 [+]로 추가(추가한 지역은 기억됨→다음에도 탭에 남음).
3. **카테고리로 좁히기 (보조)**: [공원] → **현 전체 공원** 평점순. ← 사각지대 해결.
4. **조합 가능**: [우에노]+[카페] = 우에노 카페만.

**검색 두 개 분리 (구멍 방지):** "지역 추가 검색"([+] 눌렀을 때, 지역만)과 "장소 검색"(메인 검색창, 장소만)은 별도. 같은 창이면 "이케부쿠로"를 지역으로 알지 장소로 알지 헷갈림.

**넷카페·노래방 등 검색어 팁:** MVP는 일본어 기준이라 검색도 일본어가 정확(예: ネットカフェ/漫画喫茶). 형은 일본 거주라 OK. 다국어(한국어로 쳐도 일본어로 변환 검색)는 버전2.

**왜 카테고리 입구가 필수인가 (이번에 발견한 설계 결함):**
- 문제: 지역 탭에만 의존하면, 제작자가 **안 적은 지역이 사각지대**가 됨. 예) 유저가 "도쿄" 골랐는데 타치카와 버튼이 없으면 → 쇼와기념공원(타치카와 소재)을 못 찾음. "안 적은 지역 = 유저에겐 없는 지역"이 되어, **앱 목적(발견 도움)과 정면 모순.**
- 해결: **카테고리 입구.** 타치카와 버튼이 없어도 [공원] 누르면 도쿄 전체 공원에서 쇼와기념공원이 뜸. 지역 목록의 한계를 카테고리가 메움.
- → 지역 탭 = "자주 가는 곳 바로가기(편의)". 진짜 발견 = 카테고리·검색이 담당.

**검색 범위 = 현 안 + 평점순:**
- 화면 1.5에서 고른 현 안에서만 (도쿄 골랐으면 도쿄 것만, 다른 현 안 섞임)
- 평점 높은 것부터 → 쇼와기념공원처럼 좋은 곳이 위로
- **현 경계 넘고 싶으면 (20% 케이스)**: [+직접 추가]로. (예: 가나가와 데이트 중 도쿄 오다이바 넣기)

**지역 탭 = 방법 C (기본 목록 + 검색):**
- 기본 목록: 제작자가 코드에 적는 지역 리스트 (현별). 편의 기능이지 제한 아님.
- 검색: 목록에 없어도 아무 지역·장소나 추가 가능.

**카테고리 필터**: 볼거리·체험·공원·쇼핑 등 (카페·맛집은 화면3)
**가격 필터 (슬라이더)**: 슬라이더(길 A, 구글 4단계). 예산 넘는 건 목록에서 **아예 뺌** (필터).
**정렬 버튼**: 평점순(기본) / 가격순. 보이는 것들을 줄 세움 (정렬 ≠ 필터). 거리순은 없음 — 유저가 어디서 출발하는지 모르니 기준점이 없음. (화면3은 담은 장소 기준이 있어 "도보 N분" 표시되지만 그건 표시지 정렬 아님)
  - 슬라이더(필터)로 "너무 비싼 건 빼고" + 정렬로 "그 안에서 싼 순" → 예산 안에서 싼 것부터. 둘 다 있으면 강력.

**사용 흐름 = 두 갈래 (A 타겟형 / B 영감형):**

- **갈래 A — 갈 곳이 정해진 경우 (타겟형)**: 지역 정함 → 검색창에 그 장소 입력 → 후보 나옴 → 카드로 확인 → [담기]. "이미 아는 걸 넣기."
- **갈래 B — 안 정해진 경우 (영감형)**: 지역만 정함 → 그 지역 후보 쫙 → **카테고리로 필터링하며 훑기** → 영감 얻음 → [담기]. "몰랐던 걸 발견하기."
- **B가 앱의 진짜 가치** (제작자 원래 문제 "아는 게 한정돼 아이디어 안 떠오름"을 푸는 게 B). A는 편의. 둘 다 같은 화면에서 자연스럽게.

**카드 상호작용 = 중간 방식:**
- 카드에 **기본 정보는 앱 안에** 표시: 이름·평점·영업시간·가격·사진 1장. → 대충 훑을 땐 앱 안에서 바로 판단.
- **자세히(리뷰·거리뷰·상세 위치)만 구글맵으로**: 카드의 "구글맵에서 보기" → 구글맵 앱 → 확인 → 돌아와서 [담기].
- 왜 중간: 방법 A(전부 구글맵으로 넘김)는 왔다갔다 번거롭고, 방법 B(전부 앱 안)는 만들 게 많음. 중간 = 훑기는 앱 안, 정밀 확인만 구글맵 → 마찰 최소 + 만들기 부담 적음.
- **카테고리 필터**: **볼거리·체험** (카페·먹는 건 화면 3으로 몰음 — 중복 방지)
- **가격 필터 = 슬라이더 (길 A)**: 모양은 슬라이더, 뒤에선 구글 4단계(price_level)로 작동. 정밀 엔화 아님(구글이 4단계만 줌). 정밀 가격은 버전2.
- **담기 + 선택 현황**: [+담기]로 장바구니처럼 모음. 여러 지역 넘나들며 담으니 "지금 뭘 담았는지"가 항상 보여야 함.
- **[+ 직접 추가]**: 구글 Places에 없는 장소(팝업스토어·축제·기간한정 이벤트·인스타서 본 신상 카페·숨은 장소)를 유저가 직접 입력해서 코스에 넣음.
  - **이름 + 위치 + 시간을 같이 받음** ← 중요. 위치는 두 방법: **①지도에서 톡 찍기 ②지명·근처역 검색**("시부야역" 치면 그 근처로). 정확한 좌표 아니어도 "대충 여기"면 충분.
  - **왜 위치를 추가 순간에 받나 (설계 원칙):** 이름만 받고 위치를 비우면, 화면3 지도(핀 못 꽂음)·화면4 동선·화면5 호텔근처 계산이 다 깨짐. "데이터는 만들어질 때 완전하게 채운다" → 추가 순간에 위치까지 받으면 이후 모든 화면이 그냥 작동. 구멍을 입구에서 막음.
  - API 불필요, localStorage 저장. 철학과 일치(유저 주도).
  - ※ **인스타 해시태그 연동은 뺌**: 메타가 해시태그 검색 API를 막아놨고(데이터 벽), 인스타 게시물엔 구조화된 장소 데이터도 없음. 대신 유저가 인스타서 본 걸 "직접 추가"로 옮겨 담으면 같은 니즈 충족.
- **여기선 장소만** 담음 → [다음] → 맛집 화면

### 화면 3 — 맛집 선택 (지도식, 카테고리 먼저 진입)
```
어느 장소 근처? (탭)
[🏮 가미나리몬] [🌳 국립과학박물관]   ← 화면2에서 담은 장소들
──────────────────────────
🔍 [검색: "우에노 오므라이스"...]      ← 검색창 (화면2 재활용)
──────────────────────────
🗺️ 지도: 선택 장소 📍 (처음엔 핀 없이 깨끗)
   "📍 가미나리몬 근처 · 종류를 골라줘"
──────────────────────────
🍜라멘 🍣스시 🍖야키니쿠 ☕카페 🍰디저트 🍺이자카야
   ↑ 카테고리 누르면 → 그 종류만 핀+목록 표시
──────────────────────────
[가격 ▾]                        [건너뛰기 ›]
──────────────────────────
🍢 로바타 후쿠와라이 ⭐4.5·도보3분·¥¥ [+담기]
   ...쭉 (선택한 종류만, 지도와 연동)...
──────────────────────────
   담은 맛집 1곳 → [코스 완성]
```
- **위치 기준이 화면 2와 다름**: 지역 탭이 아니라, **화면2에서 담은 장소가 기준.** 장소 카드 탭하면 그 장소가 지도 중심 → 근처 맛집이 핀+목록으로.
- **★ 카테고리 먼저 진입 (B안, 핀 지옥 방지)**: 처음 열면 지도 깨끗 + 카테고리 버튼만. 종류(라멘/카페...)를 고르면 **그 종류만** 핀+목록으로 표시. 이유: 근처 "맛집 전부"를 처음부터 다 띄우면 핀 수십 개로 지도가 뒤덮여 오히려 못 고름. "펼치는 것"과 "고를 수 있게 하는 것"은 다름 — 데이트에서 "라멘 먹을까 카페 갈까"처럼 종류부터 정하는 사고 흐름과도 일치. 부수 효과: 눌렀을 때만 API 요청 → 구글 API 호출 절약.
- **검색창 추가**: "우에노 오므라이스"처럼 직접 찾고 싶을 때. 구글 Places 텍스트 검색 사용 — 화면 2의 검색을 그대로 재활용 (새 개발 없음).
- **기술 구현**: 카테고리 = 구글 Places API의 type/keyword 파라미터로 종류 지정 요청 → 결과를 state에 → map으로 핀 그리기. V1에서 충분히 구현 가능.
- **왜 지도식인가**: 맛집은 "내가 고른 장소 근처"가 핵심. 목록만으론 "어느 장소 근처인지" 안 보임. 지도로 보면 동선 감·거리(도보 몇 분)가 직관적. (화면 5 호텔과 같은 구조 = 재활용·일관성)
- **지도 + 목록 병행**: 지도로 "어디 있는지", 목록으로 "평점·가격 비교". 핀 누르면 목록에서 강조.
- **[건너뛰기] 가능**: 맛집이 모든 데이트에 필수 아님 (예약함/타베아루키/밥 계획 없음).
- **여러 개 담기**: 후보 모으기. 최종 배치·취사선택은 화면 4에서.

### 화면 4 — 완성 (편집 가능한 타임라인) ★ 또 하나의 심장
- 단순 "보기"가 아니라 **편집 화면.** 화면 2·3에서 담은 후보들을, 화면 1의 시간(예: 12~23시)에 맞춰 타임라인에 배치.
- 유저가 **드래그앤드롭으로 순서 바꾸고, 버리고, 시간 조정.** ("두 개 이상 담고 나중에 배치하거나 버린다" = 이 화면이 그걸 함)
- 담은 게 후보라서, 여기서 "이건 넣고 이건 빼고"를 최종 결정.
- 완성된 코스 = 동선 + 시간표.
- **카드 메모**: 각 카드(장소·맛집) 클릭 → 메모 입력 ("3시 예약함", "출구 A로", "여기서 선물"). 구글 데이터엔 없는 유저만의 계획·맥락. localStorage에 memo 필드로 저장. (MVP)
- **공유 (MVP)**: 완성 코스를 **이미지/텍스트로 내보내기** → 카톡/라인으로 보냄. 백엔드 없이 "여친에게 보여주기" 가능.
- **무거운 공유 (버전2)**: 여친도 앱에서 열어 같이 보고 수정. 백엔드 필요.
- **저장**: localStorage. **단, 저장 로직은 한 군데로 추상화**(saveCourse/loadCourse 함수) → 나중에 백엔드 전환 시 함수 속만 교체, 앱 나머지는 안 건드림.
- **구글맵 경로 보기**: "구글맵에서 보기" 버튼 → 담은 장소들이 순서대로 찍힌 코스가 **구글맵 앱에서** 열림 (경로·교통수단·이동시간은 구글맵이 처리).
  - MVP = **방법 2 (구글맵 앱으로 링크)**: 형 앱은 링크만 생성, 실제 경로·교통은 구글맵이 함. 초보에게 쉬움. 단 앱을 잠깐 벗어남.
  - 나중 = 방법 1 (앱 내 지도, Google Maps JS + Directions API): 앱 안에서 경로가 그려짐. 이 대화에서 본 그 지도 표시. 여유 될 때 업그레이드.

**→ 화면 3에서 "여러 개 담기"로 정했기 때문에 화면 4가 "편집 타임라인"이 됨.** 만약 화면 3에서 확정 고르기였다면 화면 4는 단순 보기였을 것. 앞 결정이 뒤 화면 성격을 정하는 예.

### 화면 5 — 호텔 추가 (선택적 애드온)
- 화면 4에서 **[호텔 추가] 버튼** → 이 화면으로 (원하는 사람만).
```
🗺️ 지도
    📍 마지막 코스 장소 (지도 중심)
    🏨 🏨 근처 호텔들 (핀)
──────────────────────────
🏨 호텔A ⭐4.2 [+추가]
🏨 호텔B ⭐4.0 [+추가]
──────────────────────────
   고르면 → 코스 마지막 일정으로 추가
```
- **왜 화면 4 다음인가 (호텔은 위치 종속적):** 맛집·볼거리는 코스 짜기 전 골라도 되지만(동선을 그것들로 만듦), 호텔은 **데이트가 어디서 끝나는지 정해져야** 그 근처를 고를 수 있음. 시부야서 끝나면 시부야 호텔. → 코스 완성 후가 논리적으로 맞음. (화면 3.5에 넣으면 아직 동선 미정이라 "어디 근처 호텔"인지 모름)
- **지도 중심 + 근처 호텔 방식:** 마지막 장소가 지도 중심, 그 주변 호텔이 핀으로. "걸어서 갈 만한지" 거리감이 직관적 (호텔은 "끝난 곳에서 가까운지"가 핵심). Google Maps API로 구현. **화면 3(맛집)과 완전히 같은 지도 구조 → 화면·코드 재활용, 유저도 한 번 익히면 둘 다 동일.**
- **데이터 (다른 장소와 다름 — 주의):**
  - MVP: 호텔 **위치·평점·이름만** (구글 Places). 카드 누르면 예약앱/구글맵으로 → 실제 가격·예약은 거기서. **앱이 가격을 안 보여주므로 "가격 틀림" 문제 원천 차단** (제작자가 제일 아파한 문제).
  - 버전2: 예약 API(affiliate) 붙여 실시간 가격·빈방·예약까지.
  - 이유: 호텔은 **가격·빈방이 실시간으로 변함.** 구글 Places는 이걸 못 줌. 카페·볼거리(정보 고정)와 근본적으로 다른 데이터 성격.
- **선택적 애드온:** 기본 흐름(장소·맛집·완성)은 그대로, 호텔은 원하는 사람만 추가. 호텔 자주 가는 사람은 누르고, 아니면 안 눌러도 됨.
- **호텔 고르면 → 화면 4로 복귀, 코스에 카드로 추가:** 호텔도 다른 장소·맛집과 똑같은 타임라인 카드가 됨. 메모 달 수 있고(예약정보 등), 드래그로 위치 조정 가능. 일관성.

**저장된 코스도 언제든 재편집 (완성 = 잠금 아님):**
- 데이트 계획은 계속 바뀜 (여친 사정으로 못 감·날씨·예약 실패·시간 변경). 그래서 저장 후에도 편집 가능해야 함.
- **편집 = 추가 / 삭제 / 드래그 이동:**
  - 삭제: 카드 밀거나 꾹 눌러서
  - 이동: 카드 드래그로 시간대 옮김
  - 추가: **[+ 장소·맛집 추가] → 화면 2로** (검색·카테고리로 여러 개 담고 돌아옴), **[🏨 호텔 추가] → 화면 5로**
- **"추가"는 종류를 미리 안 고름**: [추가] 누르면 그냥 화면 2가 열리고, 거기서 맛집·카페·장소 뭐든 여러 개 담아서 돌아옴. 처음 코스 짤 때와 똑같은 방식 → 배울 것 없고 일관됨. (호텔만 위치 종속이라 화면 5로 예외)
- **화면 재활용**: 편집용 화면을 새로 안 만듦. 기존 화면 2·5를 다시 여는 것 = 편집. 만들 것 줄고 일관성.
- **공유와 연결**: 여친이 "여기 못 가" 하면 → MVP는 형이 수정해서 다시 이미지 공유 / 버전2는 여친이 직접 편집(무거운 공유).

**참고 — 넷카페·노래방·오락실 등 "체류형" 장소는 별도 처리 불필요:** 세 입구 구조(검색·카테고리) 덕에 구글에 등록된 곳은 검색으로 다 나옴. 개별 장소 유형을 따로 챙길 필요 없음. (좋은 구조가 "뭐 추가하지" 고민을 없앰)

**데이터 정리 (뭘 만들고 뭘 구글이 주나):**
- 지역 기본 목록 = **제작자가 직접 적는 리스트** (유일하게 만드는 데이터)
- 지역 검색 = 구글 Places (지역명→좌표)
- 장소·맛집 후보 = 구글 Places 실시간
- 가격·영업시간·평점 = 구글 Places 그대로
→ 형이 만드는 데이터는 사실상 "지역 이름 리스트" 하나뿐. 나머지는 구글이 줌.

**설계 원칙 (이번에 체득):** UI(보이는 것)와 데이터(뒤에서 되는 것)를 항상 같이 본다. "슬라이더로 하자"(UI)가 구글 데이터로 뒷받침되는지 확인 → 안 되면 길 A처럼 조정. 못 만드는 걸 그리지 않기.
