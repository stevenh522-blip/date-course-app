### 내 질문과 이해 과정

**Q. "placeId는 코드 작성자가 임의로 정한 이름 아니야?"**
- 설명: 이름표(placeId)는 우리가 정하지만, 값("ChIJ...")은 구글이 전 세계 장소에 붙인 고유번호. 도서관 청구기호 비유 — 번호를 내가 만든 게 아니라 도서관(구글)이 붙인 걸 받아적는 것.
- 이해: "아 이해됐다. lat도 마찬가지지" — 이름은 우리 것, 값은 구글 것이라는 구분을 스스로 다른 필드(lat)에 적용함.

**Q. "코스아이템에 시간이 이미 있는데 왜 Course에 또 startTime/endTime이 들어가?"**
- 설명: Course의 시간 = 데이트 전체 틀(12~23시, 화면1에서 정한 것). CourseItem의 시간 = 그 안에서 각 장소의 배치(13시...). 회의 비유 — 전체 회의 9~18시 vs 각 세션 시간.
- 이해: 같은 이름이 두 층에 있는 건 중복이 아니라 "전체 vs 개별"의 층 차이.

**Q. "즐겨찾기 장소들 이해가 안 돼" → "Place 전체에 코스 장소도 있는데 즐겨찾기에 다 섞이는 거 아닌가?"**
- 설명: `type Place`는 창고가 아니라 **설계도(붕어빵 틀)**. 실제 데이터는 favorites와 각 course의 items에 **각자 따로** 저장됨. "모든 Place 창고"는 존재하지 않으므로 섞일 수 없음. 주소록 비유 — 연락처 양식(틀)은 같지만 가족 그룹/회사 그룹에 따로 저장.
- 이해: **"아 타입이라는 게 붕어빵 틀이구나. 그걸 배열로 저장한다는 거지"** — 타입=틀, 배열=담는 상자라는 핵심 개념을 스스로 요약. 이후 계속 이 비유로 사고함.

### 배운 설계 원칙
- 시간은 장소(Place)가 아니라 배치(CourseItem)에 — 같은 장소가 여러 코스에 다른 시간으로 들어갈 수 있으므로 (배우/배역 비유)
- 예정/다녀온은 저장하지 않고 날짜로 판단 — 상태 저장하면 매일 갱신 필요, 날짜 비교면 자동
- 호텔은 items 밖에 별도 (화면 설계와 데이터 구조의 일치)

## Day 2 — HTML → CSS → JavaScript → React 개념 완주

### HTML

**Q. "footer 이런 거 들어봤는데 뭐야? span에 대해 더 알려줘"**
- 설명: footer/header/nav는 div와 기능이 같은 상자인데 "의미(이름표)"가 더해진 것. span은 div와 달리 글자만큼만 차지.
- 이해: 의미 태그의 존재를 알게 됨.

**Q. "세로로 쌓인다, 가로로 이어진다가 뭐야"**
- 설명: 그림으로 확인 — div는 한 줄 통째로 차지(전세)라 아래로 쌓이고, span은 글자만큼만(월세)이라 옆으로 이어짐.
- 이해: 이미지 보고 즉시 납득. 이후 "카드는 div, 글자 강조는 span" 기준으로 판단.

**Q. "그럼 span 필요 없이 div 하나로 flex 배치하면 되는 거 아냐?"**
- 설명: 기술적으로 절반 맞음. 근데 ① 글자 흐름 속 강조는 span이 자연스럽고 ② div만 100개면 나중에 코드를 못 읽고 ③ 접근성 문제.
- 나의 결정: **"정석대로 가줘. 남들한테 설명하는데 div만 쓸 수 없잖아"** — 발표회에서 코드 보여줄 것을 고려해 시맨틱 태그로 가기로 스스로 선택.

**Q. "section은 뭐야? 왜 h3나 p를 쓰고 span 안 쓰고? nav는 버튼 추가하면 자동으로 나눠주나?"**
- 설명: section=주제 있는 구역. h3/p/span은 "독립된 제목/문단이냐, 글자 속 일부냐"로 구분. nav는 자동 배치 안 됨 — 4등분은 CSS(flex:1)가 함.
- 이해: **"태그=의미, CSS=모양"**이 반복되는 핵심임을 파악.

**Q. "nav는 footer에 안 감싸도 돼?"**
- 설명: footer는 위치가 아니라 역할(마무리 정보). 앱 하단 탭바는 순수 메뉴라 nav 단독이 정석.
- 이해: "위치로 태그를 정하는 게 아니라 뜻으로 정한다"를 잡음.

**Q. "nav 위치가 어떻게 자동으로 맨 아래가 된 거야? main은 뭐야?"**
- 설명: 자동 아님 — 코드 순서(마지막에 써서). 화면 바닥 고정은 CSS(position:fixed; bottom:0). main = header/nav 뺀 핵심 본문.
- 이해: 방금 배운 "태그는 위치 안 정함"과 모순돼 보여서 질문 → 코드 순서 + CSS라는 답으로 해소. 앞뒤 연결하며 배우는 패턴.

**Q. "div 박스 크기는 무한이야?"**
- 설명: 가로 = 부모만큼 꽉(제한 있음), 세로 = 내용만큼. 내용이 화면 넘으면 스크롤. 고무 상자 비유.
- 이해: 크기가 "내용과 부모"로 정해진다는 것.

**Q. "그럼 왜 다 div에 집어넣은 거야?" / "nav를 div 밖으로 빼도 되지 않나?"**
- 설명: React는 하나만 return 가능 → 여러 개는 봉투(div)로 묶어야. nav를 빼도 결국 `<>`로 다시 묶어야 하고, 바깥 div가 화면 전체 배치를 관리하므로 유지가 나음.
- 이해: **"리액트로 한꺼번에 하나의 컴포넌트로 한 번에 보여준다 이 말이구나"** — 화면 하나 = 컴포넌트 하나를 스스로 정리.

### CSS

**복습 확인**: background(배경) / color(색) / padding(**"박스 안 완충재"** — 내 표현) / font-size(글자크기) / border-radius(박스 둥글게) — 다음날 5개 전부 기억.

**Q. (D-day 얘기 중) "날짜 지나면 D-가 하나씩 주는 건 HTML이 아니지? 자동으로 날짜 선택하면 D-가 생기는 기능이랑"**
- 설명: 정확함 — HTML은 죽은 글자, 계산·변화는 JavaScript. 웹 3요소: HTML(뼈대)+CSS(모양)+JS(기능).
- 의미: 배우지 않은 경계(정적 vs 동적)를 스스로 감지한 질문.

**Q. "span이면 그 span 안의 작은 박스를 꾸미는 거 아냐? 어떻게 자동으로 글자가 인식돼?"**
- 설명: 속성이 두 종류 — 글자용(color, font-size, font-weight)은 글자로, 박스용(background, padding, border-radius)은 박스로 감. 속성 자체가 목적지를 가짐.
- 이해: **"아 CSS 코드에 따라 다르구나"** — 속성별로 적용 대상이 다름을 파악. D-16 뱃지 만들기(color→background→padding→border-radius 쌓기)로 확인.

### flexbox (깊이)
- justify-content = 주축(가로) 정렬, align-items = 교차축(세로) 정렬 — "이해됐어 그 뜻이구나"
- flex-direction: row/column, gap, flex:1(공간 균등 분배 — 탭바 4등분의 비밀)
- 이전 질문("nav 자동 4등분 되나?")의 답이 flex:1이었음을 확인

### JavaScript 6벽돌

1. **변수**: let(바뀜)/const(고정) — "우리 데이트는 const, 남은 날짜는 let" 정답
2. **값 종류**: string/number/boolean — "글자/숫자/참거짓" 정답
3. **계산·연결**: `"담은 곳 " + 4 + "곳"` — 정답. 공백은 따옴표 안에 쓴 그대로 나온다는 정밀함 추가
4. **판단(if)**: rating >= 4.0 → "따봉" 정답. `===`(비교) vs `=`(담기) 구분
   - **내 연결 질문: "이거 배우니까 그럼 데이트 날짜 지나면 자동으로 이력으로 넘어가나?"** — if를 배우자마자 앱 동작(예정/다녀온 자동 분류)과 연결. "옮기는 게 아니라 볼 때마다 판단"이라는 구조 이해.
5. **반복(for)**: for...of. "선물 예약/목도리 사기/케이크 주문" 3회 출력 정답
   - **내 질문: "반복 좀 더 구체적으로"** → 슬로우모션(바퀴마다 상자 값이 덮어써짐) + 반복 안에서 계산·판단 조합 실행 확인
   - **내 발견: "자바 for문이랑 조금 다르게 생겼네"** → 자바 향상된 for(`:`)와 JS for-of(`of`)가 같은 개념, 기호만 다름
6. **함수**: function 이름(매개변수){return} — 자바 메서드와 같음(타입 생략)
   - **내 질문: "매개변수는 밖에다 만들어놓고 시작하는 거지?"** → 아님. 매개변수=함수 괄호 안 빈 상자(택배함), 인자=부를 때 넣는 값(택배). 밖의 변수든 직접 값이든 상관없음.
   - **내 질문: "인자가 필요한 함수면 변수에 어떻게 담아?"** → 담을 땐 인자 없이 함수 자체만(기계 사두기), 부를 때 인자(재료 넣고 돌리기)
   - **내 발견: "자바스크립트는 함수 자체를 변수로 설정할 수 있구나"** — 자바와의 진짜 차이를 파악. React onClick의 복선.

### React 5핵심

**컴포넌트**: 화면을 return하는 함수. "컴포넌트는 하나의 큰 함수 안에 여러 구조가 섞인 건가?" → 아님, **작게 쪼갠 함수들의 조립**. 한 컴포넌트 = 한 역할.
- 최종 이해: **"함수(=컴포넌트) 안에 여러 컴포넌트들이 합쳐진 걸 화면에 보여주는 거잖아"** — 컴포넌트 안에 컴포넌트(레고 안에 레고)

**JSX**: "자바스크립트 안에 HTML, HTML 안에 자바스크립트가 가능해진 거네" — 정확한 자기 정리
- **내 질문: "JSX가 없으면 어떻게 되는데?"** → React.createElement 지옥 확인. JSX = 편의 문법
- **내 질문: "리액트도 없이 HTML/CSS/JS만 있으면 어떻게 합치냐?"** → link/script로 연결 + getElementById로 DOM 조작. 복잡해지면 지옥 → React의 존재 이유
- 최종 이해: React는 HTML과 JS를 한 곳에서 공유(JSX) + 데이터 바꾸면 화면이 자동으로(state) → "찾아서 바꾸기"를 없앰

**props**: 처음에 "뭔 말인지 모르겠어" → 붕어빵 틀 비유로 재설명 → "건네주는 건 어떻게 해?"(컴포넌트 옆에 이름="값", HTML 속성과 같은 모양) → "건네받은 건 어떻게 써?"(function(props), props.이름) → 보내는 쪽/받는 쪽 코드 쌍으로 확인
- **내 질문: "즐겨찾기 화면이 받는 쪽 아닌가?"** → 데이터를 가진 쪽이 보내는(나눠주는) 쪽. 부모=쓰는 쪽=보냄, 자식=쓰이는 쪽=받음. 사장/직원 비유.
- **내 이해: "부모가 자식한테 보내주고 자식은 카드 만들어서 다시 보내주는 느낌"** → "다시 보내는" 게 아니라 그 자리에 화면이 됨(단방향). 
- **내 질문: "화면엔 카드 컴포넌트 하나만 보이잖아"** → 틀은 하나지만 쓴 만큼 화면에 나옴(붕어빵 틀 하나로 여러 개)
- **내 결론: "그러면 정보를 for문으로 보내면 되는 거 아냐?"** — props+반복을 스스로 조합. 정확히 React의 방식(map).

**map**: **내 질문: "왜 굳이 for을 안 쓰는지"** → for는 명령(값 안 돌려줌), map은 결과 배열을 돌려줌 → JSX {}에는 값이 필요.
- **내 요약: "for은 리턴을 한 번 더 해야 하는데 map은 그냥 바로 리턴까지 해준다"** — 정확한 본질 요약.

**state**: useState 기본. 
- **내 질문 3연발: "onClick 함수는 state야?"(아님—동작과 값의 구분) / "let인데 왜 뒤에 []가 와?"(구조 분해—두 개를 순서대로 나눠 받기) / "이미 담은 건 또 못 담잖아"(if(includes)로 중복 검사—실제 앱 로직의 허점을 스스로 발견)**
- **내 질문: "그 (set)함수는 어디 있는데?"** → useState가 자동으로 만들어 줌. TV 사면 리모컨 딸려오듯.
- **내 질문: "React가 두 개를 준다는데 어디다 주는 거야?"** → `=` 왼쪽에 내가 준비한 변수에. 오른쪽 먼저 계산 → 왼쪽에 담김.
- **내 오해와 교정: "[초기값, 변한값]인가?"** → 아님. [값 하나, 그 값을 바꾸는 함수]. 값은 하나고 그게 변하는 것.
- 버튼의 `>+1<`(표면 글자)과 onClick의 `숫자+1`(계산)은 별개 — "+1이 두 번 쓰여있는데?" 질문으로 확인
- 최종: 카운터 예제 완전 이해

## Day 3 — 강의 병행 + 설계 리뷰 + 커리어 전략

### 강의 보며 나온 질문들

**Q. "className이 뭐야? 클래스 네임으로 이름 정해놓고 CSS 파일에서 불러오네"**
- 설명: className = 태그에 이름표 → CSS 파일의 `.이름표 { }`가 찾아서 꾸밈. style={{}} 직접 쓰기와 결과 같고 방식만 다름. 실전 정석은 className+CSS 분리. HTML의 class가 React에선 className(JS 예약어 충돌).

**Q. "id는 뭐야?"**
- 설명: 하나만의 고유 이름표(주민번호) vs className(반 이름, 공유 가능). CSS에서 #이름 vs .이름. React에선 id 쓸 일 적음(찾아서 조작을 안 하니까).

**내 발견: "강의하는 사람은 그냥 div로 다 감싸네. 우리는 header, section처럼 여러 코드 쓰는데"**
- 강의(div만)도 작동하지만, 우리 방식(시맨틱)이 한 단계 위. 강의 볼 때 "저 div는 우리 식으로 뭐지?" 번역하며 보기로.

**Q. "state를 쓰면 좋은 게 뭐야? state 안에 배열 쓸 수 있어?"**
- state의 가치 = 값 바꾸면 화면 자동 갱신(렌더링). 배열 state 가능 — 단 push 금지, 새 배열로 set.
- **"배열 state 이해가 안 돼"** → 숫자 state와 같은 구조로 재설명: set숫자(숫자+1) = set목록([...목록, 새것]). 둘 다 "기존 것으로 새 값 만들어 set". `...` = 상자 열어 내용물 쏟아붓기.
- **확인 문제 정답: `[...목록, "공원"]`** — 스스로 패턴 적용 성공.

**Q. "변수 두 개 써서 useState랑 똑같은 효과 낼 수 있지 않나?"**
- 실험적으로 확인: 일반 변수 2개로는 ① React한테 알리는 벨이 없어 화면이 안 바뀌고 ② 리렌더링 시 함수가 재실행되며 값이 리셋됨.
- useState = React의 금고(함수 밖 보관, 리렌더링에도 유지) + 벨(set 호출 시 리렌더링 트리거).
- **내 요약: "렌더링을 자동으로 해주고, 함수 밖에 데이터를 저장해서 꺼낼 수 있게 하고"** — 정확.

### 내 질문과 이해 과정

**개념: `linear-gradient` (새 개념)**
- 설명: `background`는 단색뿐 아니라 그라데이션도 받는다. `linear-gradient(방향, 시작색, 끝색)`. 자리는 그대로, 넣는 값만 바뀜 — 예전 className 값 갈아끼우던 것과 같은 감각.

**개념: span 과녁 복습 (1-2)**
- 한 줄 안에서 부분만 다른 스타일 → 그 부분을 `<span className=...>`으로 떼어내 CSS 조준. (Day 2·3에서 잡은 "부분 강조는 span" 재확인)

**개념: 삼항으로 className 갈아끼우기 (1-3, 이번의 핵심)**
- 고정 클래스는 `className='dday'`(따옴표), **바뀌는** 클래스는 `className={todo.done ? "todo-done" : ""}`(중괄호=JS 표현식). map이 돌 때 항목마다 done 값이 달라 서로 다른 class가 붙음.

- 한 span 안에 삼항 두 개(`className={done? ...}` = 옷 고르기 / `{done? "☑":"☐"}` = 내용 고르기)가 역할이 다름을 구분.

### 남은 감각
- 색·각도는 "외우는 게 아니라 원본에서 고르는 것" — 스케치 소스가 정답지.
- 화면 일부만 다르게 = span. 상황 따라 달라지면 = className을 삼항으로.

## Day 5 — Phase 1-4~1-6 (flex 중첩 실전, 하루에 3개)

> 기준: `docs/sketches_final/screen0_home_FINAL.png`. 세 작업 다 결국 **"flex 부모를 어디에 두느냐"** 한 문제였다.

### 한 것
- **1-4. 탭바 2층**: 아이콘/글자를 span으로 쪼개고 `flex-direction: column` + `gap:4px`. 홈만 `active` 핑크
- **1-5. 영역별 ＋ 버튼**: h3에 `display:flex` + `justify-content: space-between` (제목 왼쪽 · ＋ 오른쪽)
- **1-6. 코스 카드**: 🌸 / 내용 div / › 를 직계 자식 3개로. `align-items:center` + 내용에 `flex:1`
- 가짜 데이터에 `startTime`/`endTime` 추가 (데이터설계도 `Course` 타입에 맞춤), `region` 제거
- 커밋+푸시 2건 (`57cc2db`, `9504bb6`)

### 내 질문과 이해 과정

**내 발견(중요): "홈만 핑크인 이유는 홈화면이라 그런 거 아닌가? 나중에 홈 화면이 아닐 때는 핑크가 아닐 거 아냐"**
- 하드코딩하라는 지시를 받고 **그 값이 임시라는 걸 스스로 간파.** 핑크는 "홈 버튼의 색"이 아니라 "지금 보고 있는 화면의 탭 색"이다.
- 그래서 이름이 `home-pink`가 아니라 `active`여야 함 — 나중에 어느 탭에 붙어도 말이 되는 이름. 지금 홈에 박아둔 `active`는 **가짜 데이터와 같은 성격**(모양 확인용 임시값).
- 이해: 이름을 잘 지어두면 1-7·라우팅 이후 **CSS는 한 글자도 안 고치고** JSX의 박아둔 값만 삼항으로 바꾸면 끝난다. (Day 4의 "삼항으로 className 갈아끼우기"가 여기로 이어짐)

**Q. "jsx 쪼개기가 뭐야"**
- 설명: `<button>🏠홈</button>`에서 🏠와 홈은 **글자 한 덩어리**라 따로 조준이 불가능. `font-size:24px`를 주면 둘 다 24px이 됨.
- 연결: **Day 4의 D-16 span 과녁과 같은 상황.** 그때는 조각 하나(D-16)만 과녁이 필요했고, 이번엔 두 조각 다 필요할 뿐.
- 이해: 이름은 `tab-icon` / `tab-label`로 지음 — "일주일 뒤의 내가 알아볼 이름"(span1/span2 금지).

**내 질문(스스로 답을 맞힘): "gap을 넣으면 버튼끼리가 아니라 버튼 안에 있는 자식요소한테 가는 건가"**
- 정확함. **`gap`은 그 규칙이 조준한 요소의 "직계 자식들 사이"에 들어간다.**
  - `.tab-bar`에 gap → 자식은 button 4개 → 버튼끼리 벌어짐
  - `.tab-bar button`에 gap → 자식은 span 2개 → 아이콘↔글자 사이가 벌어짐
- **"직계"가 핵심 — gap은 손자에게 안 흘러간다.** 각 flex 부모가 자기 자식만 책임진다.
- 파생 이해: **button의 이중 신분** — 위에서 보면 `.tab-bar`의 자식, 아래에서 보면 span 2개의 부모. 이게 flex 중첩의 전부.
- (버튼끼리 간격은 gap이 아니라 `flex:1`이 처리 중 — 4개가 남는 폭을 균등하게 나눠 갖고 각자 자기 칸 안에서 가운데 서기 때문. Day 2의 "탭바 4등분의 비밀"이 여기서 재확인됨)

**개념: 우선순위(specificity) (새 개념)**
- `.tab-bar button`에 이미 `color:#8a8a99`(회색)가 있는데 `.active`로 핑크를 주면 **누가 이기나?**
- 규칙: **클래스 개수를 먼저 세고, 같으면 태그 개수를 센다.**
  - `.tab-bar button` = 클래스 1 + 태그 1
  - `.active` = 클래스 1 + 태그 0 → 클래스 1:1 비김 → 태그 0:1로 **짐** (회색 유지)
  - `.tab-bar .active` = 클래스 2 → 첫 판에서 **이김** ✅
- 두 번째 이유(스코프): `.tab-icon`처럼 접두사 없이 두면 다른 화면이 같은 이름을 쓸 때 참견함. **부품의 CSS는 그 부품 안에서만 논다** → `.tab-bar .tab-icon`으로 정정.
- 적용: 1-5에서 시키지 않았는데 `.my-note .plus-button`으로 스스로 접두사를 붙임.

**개념: 주축 vs 교차축 (오늘 제일 중요)**
- `flex-direction`을 바꾸면 `justify-content`와 `align-items`의 **의미도 같이 뒤집힌다.**

| flex-direction | 주축 | justify-content | align-items |
|---|---|---|---|
| `row`(기본) | 가로 → | 좌우 | 위아래 |
| `column` | 세로 ↓ | 위아래 | 좌우 |

- **외울 것은 방향이 아니라 규칙 하나:** justify는 언제나 주축, align은 언제나 교차축. 주축이 어디인지는 `flex-direction`이 정한다.
- 오늘 실제로 둘 다 씀 — 탭바 button은 `column`(주축=세로), 코스 카드는 `row`(주축=가로). 방향으로 외웠으면 반드시 틀렸을 자리.
- (Day 2에서 "justify=가로, align=세로"로 이해했던 것의 **정정·심화**. 그건 `row`일 때만 맞는 말이었음)

**내 막힘: "구조 부분은 잘 모르겠어" (1-6, 오늘의 최대 난관)**
- 1-4·1-5는 flex 하나였는데 1-6은 **flex 안에 flex** + `flex:1` + `align-items`가 한꺼번에 나옴. 게다가 "화면에 안 보이는 div를 왜 만드나"가 처음이었음.
- 단계별로 화면이 어떻게 보이는지로 해소:
  1. flex 켜기 전 → h3/p가 block이라 저절로 세로로 쌓임 (브라우저 기본 동작)
  2. `display:flex`만 켜면 → **제목과 부제가 옆으로 나란히** 서버림 (flex가 자식의 원래 성질을 덮어씀)
  3. 🌸·›를 그냥 추가하면 → 자식이 넷 → `🌸 제목 부제 ›` 전부 가로
- **막히는 지점:** "제목과 부제만 세로로 해달라"고 부모에게 부탁할 방법이 없다. flex 부모에겐 **"자식 전부 가로" 아니면 "자식 전부 세로"**뿐, 일부만 다르게는 불가능.
- **발상 전환:** ❌ "부모한테 얘네 둘만 다르게 해달라" → ⭕ **"얘네 둘을 하나로 만들어서 부모가 자식 하나로 보게 하자"**
- **묶는 div의 정체:** 화면에 아무것도 안 그리는데 왜 있냐면 — **flex의 사정거리를 끊으려고.** div로 감싸는 순간 카드의 "가로로!" 명령이 div에서 멈추고, 그 안에서는 원래 성질(block=세로)이 되살아남. (비유: div는 방문을 닫은 방. 카드는 "방 하나"를 일렬에 세울 뿐 방 안까지 명령이 안 닿음)

**개념: `flex: 1` — 남는 폭 먹기 (`space-between`과 뭐가 다른가)**
- flex 자식은 기본적으로 **자기 내용물만큼만** 폭을 차지 → 셋이 왼쪽에 오글오글 모이고 오른쪽이 텅 빔.
- 선택지 A `space-between`: 남는 공간을 **자식 사이에 뿌림.** 1-5는 자식이 **둘**이라 사이가 하나뿐 → 통째로 가운데 몰리고 둘이 양끝으로 밀림(예쁨). 근데 1-6은 자식이 셋이라 **제목이 가운데로 떠버림**(스케치와 다름).
- 선택지 B `flex:1`(채택): 가운데 div를 **뚱뚱하게 만들어서** ›를 오른쪽 끝으로 **밀어냄.** 🌸는 왼쪽에 딱, 제목은 그 바로 옆 유지.
- **한 줄 차이:** `space-between`은 **빈칸을 사이에 넣는** 것, `flex:1`은 **한 놈을 뚱뚱하게 해서 밀어내는** 것. 결과가 비슷해 보여도 원리가 정반대.
- **같은 `flex:1`인데 결과가 달랐던 이유:**

| | 누가 `flex:1`? | 결과 |
|---|---|---|
| 탭바 | 버튼 4개 **전부** | 남는 폭을 1:1:1:1로 **나눠** 가짐 → 균등 |
| 코스 카드 | div **하나만** | 남는 폭을 **혼자 다** 먹음 → 옆을 밀어냄 |

- 규칙은 하나: `flex:1` = "남는 공간 분배에 참여, 지분 1." 참여자가 넷이면 1/4씩, 하나면 독식. 🌸·›는 `flex:1`이 없어 **분배 불참** → 자기 크기 그대로.
- **같은 속성인데 "누가 갖고 있느냐"로 결과가 달라진다** — 이해했다/외웠다의 갈림길.

**개념: `align-items: center`가 필요했던 이유**
- 없으면 🌸와 ›가 **위로 붙음.** flex 자식의 기본값이 `stretch`(교차축으로 부모 높이만큼 늘어남)라서, 🌸/› span이 카드 높이만큼 늘어난 **투명한 상자**가 되고 글자는 그 상자 맨 위에 앉음.
- `align-items:center` = "늘리지 말고 교차축 중앙에 놔".

**개념: 맨몸 텍스트도 flex 자식이다**
- `<h3>🎂 기념일 <button>+</button></h3>` — span으로 안 감쌌는데 `space-between`이 먹었다. flex가 맨몸 텍스트를 **보이지 않는 상자에 담아** 자식으로 쳐주기 때문(익명 flex 아이템).
- 단, **그 텍스트만 따로 조준할 과녁은 없다** → 나중에 제목 색만 바꾸고 싶어지면 그때 감싸야 함. 지금은 안 감싸도 동작 동일.

### 오늘 걸린 것들 (실수 노트)

**1. 저장을 안 했는데 CSS를 의심했다**
- `flex-direction: column`을 넣었는데 화면이 "그냥 그대로". CSS 문제인 줄 알았으나 **JSX 미저장**이 원인. 채팅엔 붙여넣었지만 파일엔 안 들어감.
- **확인법(앞으로의 무기):** dev 서버 로그에 `hmr update /경로`가 찍혔는지 본다. CSS는 두 번 찍혔는데 JSX는 한 번도 안 찍혀 있었음 → 안 찍혔다 = 안 바뀌었다.
- 왜 "그대로"였나: CSS는 제대로 들어갔지만 button 안이 여전히 `🏠홈` 한 덩어리 → **쌓을 재료가 없었던 것.**

**2. undefined는 조용히 사라진다**
- 데이터에서 `region`을 뺐는데 JSX에 `📍{course.region}`이 남음. React는 undefined를 아무것도 안 그려서 **에러 없이 📍만 덩그러니** 남음.
- 제일 놓치기 쉬운 종류 — 터지지 않고 조용히 이상해진다.

**3. 구분점은 사이에만**
- `… · 4곳 ·`처럼 마지막에도 `·`가 붙음. **`gap`이 자식 "사이에만" 들어가는 것과 같은 감각.**

**4. Claude도 파일을 안 보면 틀린다**
- "h3/p 기본 margin 때문에 카드가 뜰 것"이라는 지적을 받았으나 **안 떴다.** `index.css`에 `* { margin: 0 }` 리셋이 이미 있었기 때문.
- 내가 이미 CSS 리셋을 해둔 덕에 `.my-note h3`의 margin도 "기본값 덮어쓰기"가 아니라 **"0에서 원하는 만큼 주기"**였던 것. 훨씬 예측하기 쉬운 방식.
- **교훈: 파일을 안 보고 일반론으로 넘겨짚으면 틀린다 — 나한테나 AI한테나 똑같이 적용.** AI 말도 한 번 의심할 것.

### 데이터 설계도 준수 (1-6)

- `startTime:"12:00"`, `endTime:"23:00"` 추가는 화면에 시간 띄우려고가 아니라 **설계도의 `Course` 타입이 원래 그렇게 생겼기 때문.**
- 가짜 데이터를 진짜와 **같은 모양**으로 만들어두면 나중에 진짜 데이터가 들어올 때 JSX를 안 고쳐도 됨. **지금 5초 쓰고 나중에 30분 아끼는 것.**
- `region`은 설계도에도 스케치에도 없어서 제거 — 있으면 나중에 "이거 왜 있지?" 하고 멈칫하게 되는 짐.
- **일부러 남긴 빚(알고 두는 것):**
  - `date:"7/12 (토)"` — 설계도는 `"2026-07-11"`. 화면용 변환은 나중에 함수로 할 일.
  - `count:4` — 원래 `items.length`로 나와야 함. items 배열이 아직 없어서 숫자로 박아둠.
- **모르고 둔 것과 알고 둔 것은 완전히 다르다.**

### 남은 감각
- 세 작업이 전부 같은 문제였다 — **flex 부모를 어디에 두느냐.** 1-4는 button을 부모로(세로), 1-5는 h3를 부모로(양끝), 1-6은 카드를 부모로 + div로 사정거리 끊기.
- 방향을 외우지 말고 **"주축이 어디인지"부터 묻기.**
- 화면이 안 바뀌면 **CSS를 의심하기 전에 `hmr update`부터.**

## Day 6 — Phase 1-9 잔불 정리 시작: key prop (map의 이름표)

> Phase 1-7(첫 state) 들어가기 전에, "1-7에서 진짜 버그로 터질 자리"라고 적어둔 key 경고부터 잡음.
> 세 컴포넌트(MyNote·CourseList·GirlfriendInfo)의 map에 이름표(key) 달기.

### 한 것
- **key 없던 map 4곳 잡기**: MyNote의 anniversaries·todos / CourseList / GirlfriendInfo
- **각 가짜 데이터에 `id` 추가**: 설계도 타입대로 (Anniversary·Todo·Course엔 이미 id가 있었고, 가짜 데이터에만 빠져있던 걸 채움)
- **설계도 갱신(A안)**: `girlfriendInfos` 타입 `{ text }` → `{ id; text }`. 여친정보도 추가·삭제되는 리스트라 id가 필요.
- 콘솔의 key 경고 전부 해소

### 내 질문과 이해 과정

**Q. "key가 뭔지 모르겠어"**
- 설명: map은 붕어빵 여러 개 찍기 → 겉보기 똑같은 형제들. React가 데이터 바뀔 때 "이전의 이 항목 = 지금 다시 그린 것 중 어느 거?"를 짝지어 **최소만 바꾸려면** 각 항목에 안 바뀌는 고유 이름표가 필요. 그게 key. 없으면 순서(index)로 때우다가 추가·삭제 때 어긋남(출석부에서 한 명 빠지면 "3번째"가 딴 사람 되는 것).
- **내 요약: "React가 화면을 정확하게 바꾸기 위해 이름표를 붙여주는 거구나"** — 핵심을 한 문장으로.

**Q. "todos.id가 아니라 todo.id네?"**
- 복수 `todos` = 통 전체 / 단수 `todo` = map이 한 바퀴마다 꺼내주는 한 개(내가 괄호 안에 지은 이름). id는 개별 항목에만 있지 통 전체엔 없음. (반 전체 vs 학생 한 명 — 학번은 학생한테)
- **map 괄호 안 이름 = 뒤에서 `.id` 붙일 때 쓰는 이름. 이 둘이 짝.**

**Q. "그럼 `key={item.여친생일}` 이런 것도 되겠네?"**
- 정정: 점 뒤엔 **값이 아니라 칸 이름**을 쓴다. `여친 생일`은 `title` 칸의 **값**. `item.여친생일`은 그런 칸이 없어 `undefined`. 값을 꺼내려면 그 값이 든 칸 이름(`title`)을 써야 함.
- 파생: `key={item.title}`은 문법상 되지만 이름표론 부적합(수정하면 바뀌고, 겹칠 수 있음) → 그래서 안 바뀌는 전용 `id` 칸을 일부러 만든다.

**Q. "스텝2(`key=`)는 뭐하는 과정이야?"**
- 스텝1(데이터에 id) = **이름표 인쇄만** 한 것. React는 아직 모름. 스텝2(`key={id}`) = 그 이름표를 **React에게 실제로 붙여줌.** 만들고(1)+붙여야(2) 완성. key는 화면에 안 보이는 **React 전용 속성**이라 모양은 안 바뀜(경고 사라지는 걸로만 확인).

**Q. "key는 가장 바깥쪽에 달아야 하는 거 아냐?"**
- 맞음. 단 "가장 바깥" = **map이 반복해 찍는 각 항목의 최상위 요소**(GirlfriendInfo에선 `<p>`). 리스트 전체를 감싸는 통(`<div className='gf-info'>`)은 한 번만 그려지는 컨테이너라 key 대상 아님.

### 설계 결정 (A안)
- 여친정보 id 필요성을 스스로 짚음: **"변경사항이 생기니까 id가 필요하다."** 추가·삭제되는 리스트는 안 바뀌는 이름표가 있어야 한다는 원리를 여친정보에 스스로 적용.
- 문서-코드 규칙(문서가 기준)대로 **설계도를 먼저 고치고** 코드를 맞춤.

### 남은 감각
- **에러는 헤매지 말고 읽기** — React가 `Check the render method of GirlfriendInfo` / `passed a child from CourseList`로 **어느 컴포넌트인지 찍어줌.** (내가 예고 못 한 CourseList 경고도 이 메시지로 바로 찾음.)
- **key 패턴은 세 곳 다 동일** = 데이터에 id(스텝1) → 반복 요소에 `key={x.id}`(스텝2).

### 한 것

  - `setSelected`를 **props(함수)로** GirlfriendInfo에 내려줌
- **1-8. 임시본 점선 카드(DraftCard 부품)**: 새 파일 `DraftCard.jsx`+`.css`. 핑크 점선(`#ffb8c9`)+✍️+"만드는 중·지역"+✕. 예정 코스 맨 위

- 커밋+푸시 3건 (`933db47`, `f1ad7e7`, `aab04af`)

### 내 질문과 이해 과정 — 1-7 (첫 state)

**개념: "무엇을 기억할까" — state에 담을 값**
- 내 답(정답): `useState(null)` — `null`=아무것도 안 눌림(시작), 누르면 그 항목 `id` 저장.
- 왜 보통 변수가 아니라 state? Day 3에서 잡은 것 재확인: 보통 변수는 바꿔도 화면 안 바뀜. state여야 "값 바뀌면 React가 다시 그림".

**정정: `usestate`→`useState`, `let`→`const`**
- `const`인데 값이 바뀌는 이유: **`selected`를 직접 대입(`selected=...`)하는 일이 없다.** 바꾸는 건 항상 `setSelected()`. 칸 자체는 안 건드리니 `const`.

**내 답: "선언 어디에? → A(return 위). 리턴 범위에 다 닿아"** — 정답.
- 심화: `useState`는 **컴포넌트 함수 안에서만** 호출 가능 → 함수 밖(C) 불가.

**내 지적(내가 Claude를 잡음): "infos도 함수 안에 있는데?"**
- Claude가 "(C) infos처럼 함수 밖"이라 잘못 예시함. infos는 함수 안이 맞음 → 지적해서 정정받음. (Day 5의 "AI 말도 한 번 의심" 실천)

**내 질문: "리턴 안에 넣어도 범위 닿지 않나?"**
- 순수 JS 범위론으론 내 감이 맞음. 근데 `return ( )` 안은 **JSX** → "문장(선언)"은 못 넣고 "값(식)"만 됨. `{todo.done ? ...}`처럼 JSX 중괄호엔 결과값 있는 것만. `const [..]=..`는 선언 문장이라 불가.

**개념: 이벤트 핸들러 `onClick` (새 개념)**
- 함정: `onClick={setSelected(info.id)}` ❌ (그리는 순간 당장 실행) vs `onClick={()=>setSelected(info.id)}` ✅. `()=>`로 감싸 **"실행 결과"가 아니라 "누르면 실행할 함수 자체"**를 건넴.

**내 헷갈림: "어느 p에 달아?"**
- `<p>`는 코드엔 하나지만 map이 데이터 수만큼 찍음 → 화면엔 2개. onClick은 그 하나뿐인 `<p>`에 달면 각 복제본이 자기 `info.id`를 안고 박힘. (Day 6 key의 "map이 하나로 여러 개" 감각)

**내 질문: "key 뒤야? key 앞이면 안 되나?"** → 속성 순서 자유 (HTML 속성처럼).

**내 확인: "첫번째 클릭하면 g1 들어가?"** → 정확. `setSelected("g1")` 실행 → `selected`가 `"g1"`.

**마지막 조각: 삼항으로 조건부 렌더**
- `{selected === info.id ? <span>✏️🗑️</span> : null}`. Day 4의 삼항 className과 같은 원리. 새 포인트 둘: ①참일 때 글자 대신 **태그**를 넣을 수 있음 ②거짓일 때 `null`=아무것도 안 그림(빈 `""`와 같은 "없음").

### 내 질문과 이해 과정 — state 끌어올리기 (오늘 두 번째 새 개념)

**원칙: state는 "그걸 쓰는 것들의 공통 부모"에 둔다**
- 여친정보 안에 갇힌 `selected`를 세 영역이 공유하려면 → 셋의 공통 부모 **MyNote**로 올림. (리모컨은 막내 방 말고 거실에)

**props로 함수 넘기기**
- 부모: `<GirlfriendInfo selected={selected} setSelected={setSelected}/>`. **`setSelected`는 함수인데 props로 넘어감** — "함수도 props로" (Day 2의 "JS는 함수를 변수로"의 실전).
- 자식: `function GirlfriendInfo ({selected, setSelected})`로 받음.
- **★ 자식의 원래 useState는 삭제.** 안 지우면 자식이 자기 리모컨을 써서 공유가 안 됨. (안 지워서 `selected already declared` 중복 선언 에러 냄)

**개념: 공유 vs 따로**
- 세 영역이 `selected` 하나 공유 = 항상 하나만 열림(사양 "그 항목에만"). 영역마다 따로 두면 최대 3개 동시에 뜸.

**내 발견: "우리 map은 return 안 썼는데?" (좋은 질문)**
- 화살표 오른쪽 `( )`(소괄호)=그냥 **묶기**라 자동 return / `{ }`(중괄호)=명령 블록이라 `return` 직접 써야 함. 우리 map은 `(info)=>( <p>... )`라 `( )`→자동 return이라 안 써도 맞음. (`{ }`였다면 화면 텅 빔 — 자주 만날 함정)

**내 질문: "돌려받을 값이 뭐야?"**
- 함수가 끝내고 손에 쥐여주는 결과물(`return` 뒤). `더하기(2,3)`→5는 결과물 있음 / `setSelected()`는 실행만, 결과물 없음. map은 돌려받은 `<p>`를 화면에 씀 → return 필요. onClick은 결과물 안 봄 → return 있든 없든 무관.

### 내 질문과 이해 과정 — 1-8 (부품) / 1-9 (대조)

**스케치가 없다는 걸 확인**
- 점선 코스 카드가 어느 FINAL 스케치에도 없음. 사양서엔 기능 확정(임시본=점선·맨 위·✕삭제)인데 그림만 없음 → 규칙(추측 말고 확인)대로 멈춤. 결국 **`add_to_course_final.html`의 임시본 행**이 기준임을 확인. 정확한 값 읽음: `border:1.5px dashed #ffb8c9`, 배경 `#fffafb`, ✍️(18px), 제목(14px/700/#3a3a4a)·부제(11px/#b0b0bd).
- ✕는 그 시트 행엔 없음(고르는 화면이라) → **홈 전용 추가**(삭제 자리).

**개념: `border` 점선 (새 개념)**
- `border: 1.5px dashed #ffb8c9` = 굵기·**종류**·색. `solid`(실선) 대신 `dashed`(점선)=미완성 느낌. 이게 1-8의 핵심. 나머지(flex·flex:1·gap)는 CourseList 카드 복습.

**margin:0 / 전역 리셋** (Day 5 재확인)
- h3/p는 기본 바깥 여백을 달고 태어남 → 카드에선 `margin:0`으로 끔. 근데 `index.css`에 `* { margin:0 }` 전역 리셋이 이미 있어 **있으나마나** → 뺐음(CourseList와 통일). `box-sizing:border-box`도 거기 있어 border 추가해도 카드 크기 안 튐.

**위치 논쟁: 완성 코스 아래로?**
- 사양은 "맨 위". 내가 "완성 코스 아래는?" 제안 → 임시본=지금 이어서 할 급한 것이라 맨 위가 맞음(최신이 위, 점선=미완성은 눈에 띄게). 사양대로 감. 바꾸려면 사양·문서부터 고치는 규칙 확인.
- 카드 간격(margin 8px)·✕ 크기는 브라우저 보고 **스스로 조정.**

**개념: 선택 항목 핑크 테두리 + CSS 우선순위 (Day 5 specificity 재등장)**
- 선택된 항목에 `.selected` 클래스(삼항). 근데 그냥 `.selected`(클래스1)만 쓰면 안 먹힘 — 기존 `.my-note p`(클래스1+태그1)가 **더 구체적**이라 배경을 못 이김. → `.my-note p.selected`(클래스2+태그1)로 써야 이김. (Day 5에서 잡은 specificity를 실전에서 또 씀)

**실수: `item.id` vs `info.id`**
- 여친정보 className에 `item.id`를 씀(기념일 줄 복붙) — 이 파일 변수는 `info` → `item is not defined` 에러. (Day 6의 "복수 통/단수 개별, map 괄호 이름이 뒤 `.id`와 짝" 재확인). 에러 메시지 읽고 바로 고침.
