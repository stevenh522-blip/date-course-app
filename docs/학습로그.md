### 내 질문과 이해 과정

**Q. "placeId는 코드 작성자가 임의로 정한 이름 아니야?"**
- 설명: 이름표(placeId)는 우리가 정하지만, 값("ChIJ...")은 구글이 전 세계 장소에 붙인 고유번호. 도서관 청구기호 비유 — 번호를 내가 만든 게 아니라 도서관(구글)이 붙인 걸 받아적는 것.
- 이해: "아 이해됐다. lat도 마찬가지지" — 이름은 우리 것, 값은 구글 것이라는 구분을 스스로 다른 필드(lat)에 적용함.

**Q. "코스아이템에 시간이 이미 있는데 왜 Course에 또 startTime/endTime이 들어가?"**
- 설명: Course의 시간 = 데이트 전체 틀(12~23시, 화면1에서 정한 것). CourseItem의 시간 = 그 안에서 각 장소의 배치(13시...). 회의 비유 — 전체 회의 9~18시 vs 각 세션 시간.
- 이해: 같은 이름이 두 층에 있는 건 중복이 아니라 "전체 vs 개별"의 층 차이.

**Q. "즐겨찾기 장소들 이해가 안 돼" → "Place 전체에 코스 장소도 있는데 즐겨찾기에 다 섞이는 거 아닌가?"**
- 설명: `type Place`는 창고가 아니라 **설계도(붕어빵 틀)**. 실제 데이터는 favorites와 각 course의 items에 **각자 따로** 저장됨. "모든 Place 창고"는 존재하지 않으므로 섞일 수 없음. 주소록 비유 — 연락처 양식(틀)은 같지만 가족 그룹/회사 그룹에 따로 저장.
- 이해: **"아 타입이라는 게 붕어빵 틀이구나. 그걸 배열로 저장한다는 거지"** — 타입=틀, 배열=담는 상자라는 핵심 개념을 스스로 요약. 이후 계속 이 비유로 사고함.

### 배운 설계 원칙
- 시간은 장소(Place)가 아니라 배치(CourseItem)에 — 같은 장소가 여러 코스에 다른 시간으로 들어갈 수 있으므로 (배우/배역 비유)
- 예정/다녀온은 저장하지 않고 날짜로 판단 — 상태 저장하면 매일 갱신 필요, 날짜 비교면 자동
- 호텔은 items 밖에 별도 (화면 설계와 데이터 구조의 일치)

## Day 2 — HTML → CSS → JavaScript → React 개념 완주

### HTML

**Q. "footer 이런 거 들어봤는데 뭐야? span에 대해 더 알려줘"**
- 설명: footer/header/nav는 div와 기능이 같은 상자인데 "의미(이름표)"가 더해진 것. span은 div와 달리 글자만큼만 차지.
- 이해: 의미 태그의 존재를 알게 됨.

**Q. "세로로 쌓인다, 가로로 이어진다가 뭐야"**
- 설명: 그림으로 확인 — div는 한 줄 통째로 차지(전세)라 아래로 쌓이고, span은 글자만큼만(월세)이라 옆으로 이어짐.
- 이해: 이미지 보고 즉시 납득. 이후 "카드는 div, 글자 강조는 span" 기준으로 판단.

**Q. "그럼 span 필요 없이 div 하나로 flex 배치하면 되는 거 아냐?"**
- 설명: 기술적으로 절반 맞음. 근데 ① 글자 흐름 속 강조는 span이 자연스럽고 ② div만 100개면 나중에 코드를 못 읽고 ③ 접근성 문제.
- 나의 결정: **"정석대로 가줘. 남들한테 설명하는데 div만 쓸 수 없잖아"** — 발표회에서 코드 보여줄 것을 고려해 시맨틱 태그로 가기로 스스로 선택.

**Q. "section은 뭐야? 왜 h3나 p를 쓰고 span 안 쓰고? nav는 버튼 추가하면 자동으로 나눠주나?"**
- 설명: section=주제 있는 구역. h3/p/span은 "독립된 제목/문단이냐, 글자 속 일부냐"로 구분. nav는 자동 배치 안 됨 — 4등분은 CSS(flex:1)가 함.
- 이해: **"태그=의미, CSS=모양"**이 반복되는 핵심임을 파악.

**Q. "nav는 footer에 안 감싸도 돼?"**
- 설명: footer는 위치가 아니라 역할(마무리 정보). 앱 하단 탭바는 순수 메뉴라 nav 단독이 정석.
- 이해: "위치로 태그를 정하는 게 아니라 뜻으로 정한다"를 잡음.

**Q. "nav 위치가 어떻게 자동으로 맨 아래가 된 거야? main은 뭐야?"**
- 설명: 자동 아님 — 코드 순서(마지막에 써서). 화면 바닥 고정은 CSS(position:fixed; bottom:0). main = header/nav 뺀 핵심 본문.
- 이해: 방금 배운 "태그는 위치 안 정함"과 모순돼 보여서 질문 → 코드 순서 + CSS라는 답으로 해소. 앞뒤 연결하며 배우는 패턴.

**Q. "div 박스 크기는 무한이야?"**
- 설명: 가로 = 부모만큼 꽉(제한 있음), 세로 = 내용만큼. 내용이 화면 넘으면 스크롤. 고무 상자 비유.
- 이해: 크기가 "내용과 부모"로 정해진다는 것.

**Q. "그럼 왜 다 div에 집어넣은 거야?" / "nav를 div 밖으로 빼도 되지 않나?"**
- 설명: React는 하나만 return 가능 → 여러 개는 봉투(div)로 묶어야. nav를 빼도 결국 `<>`로 다시 묶어야 하고, 바깥 div가 화면 전체 배치를 관리하므로 유지가 나음.
- 이해: **"리액트로 한꺼번에 하나의 컴포넌트로 한 번에 보여준다 이 말이구나"** — 화면 하나 = 컴포넌트 하나를 스스로 정리.

### CSS

**복습 확인**: background(배경) / color(색) / padding(**"박스 안 완충재"** — 내 표현) / font-size(글자크기) / border-radius(박스 둥글게) — 다음날 5개 전부 기억.

**Q. (D-day 얘기 중) "날짜 지나면 D-가 하나씩 주는 건 HTML이 아니지? 자동으로 날짜 선택하면 D-가 생기는 기능이랑"**
- 설명: 정확함 — HTML은 죽은 글자, 계산·변화는 JavaScript. 웹 3요소: HTML(뼈대)+CSS(모양)+JS(기능).
- 의미: 배우지 않은 경계(정적 vs 동적)를 스스로 감지한 질문.

**Q. "span이면 그 span 안의 작은 박스를 꾸미는 거 아냐? 어떻게 자동으로 글자가 인식돼?"**
- 설명: 속성이 두 종류 — 글자용(color, font-size, font-weight)은 글자로, 박스용(background, padding, border-radius)은 박스로 감. 속성 자체가 목적지를 가짐.
- 이해: **"아 CSS 코드에 따라 다르구나"** — 속성별로 적용 대상이 다름을 파악. D-16 뱃지 만들기(color→background→padding→border-radius 쌓기)로 확인.

### flexbox (깊이)
- justify-content = 주축(가로) 정렬, align-items = 교차축(세로) 정렬 — "이해됐어 그 뜻이구나"
- flex-direction: row/column, gap, flex:1(공간 균등 분배 — 탭바 4등분의 비밀)
- 이전 질문("nav 자동 4등분 되나?")의 답이 flex:1이었음을 확인

### JavaScript 6벽돌

1. **변수**: let(바뀜)/const(고정) — "우리 데이트는 const, 남은 날짜는 let" 정답
2. **값 종류**: string/number/boolean — "글자/숫자/참거짓" 정답
3. **계산·연결**: `"담은 곳 " + 4 + "곳"` — 정답. 공백은 따옴표 안에 쓴 그대로 나온다는 정밀함 추가
4. **판단(if)**: rating >= 4.0 → "따봉" 정답. `===`(비교) vs `=`(담기) 구분
   - **내 연결 질문: "이거 배우니까 그럼 데이트 날짜 지나면 자동으로 이력으로 넘어가나?"** — if를 배우자마자 앱 동작(예정/다녀온 자동 분류)과 연결. "옮기는 게 아니라 볼 때마다 판단"이라는 구조 이해.
5. **반복(for)**: for...of. "선물 예약/목도리 사기/케이크 주문" 3회 출력 정답
   - **내 질문: "반복 좀 더 구체적으로"** → 슬로우모션(바퀴마다 상자 값이 덮어써짐) + 반복 안에서 계산·판단 조합 실행 확인
   - **내 발견: "자바 for문이랑 조금 다르게 생겼네"** → 자바 향상된 for(`:`)와 JS for-of(`of`)가 같은 개념, 기호만 다름
6. **함수**: function 이름(매개변수){return} — 자바 메서드와 같음(타입 생략)
   - **내 질문: "매개변수는 밖에다 만들어놓고 시작하는 거지?"** → 아님. 매개변수=함수 괄호 안 빈 상자(택배함), 인자=부를 때 넣는 값(택배). 밖의 변수든 직접 값이든 상관없음.
   - **내 질문: "인자가 필요한 함수면 변수에 어떻게 담아?"** → 담을 땐 인자 없이 함수 자체만(기계 사두기), 부를 때 인자(재료 넣고 돌리기)
   - **내 발견: "자바스크립트는 함수 자체를 변수로 설정할 수 있구나"** — 자바와의 진짜 차이를 파악. React onClick의 복선.

### React 5핵심

**컴포넌트**: 화면을 return하는 함수. "컴포넌트는 하나의 큰 함수 안에 여러 구조가 섞인 건가?" → 아님, **작게 쪼갠 함수들의 조립**. 한 컴포넌트 = 한 역할.
- 최종 이해: **"함수(=컴포넌트) 안에 여러 컴포넌트들이 합쳐진 걸 화면에 보여주는 거잖아"** — 컴포넌트 안에 컴포넌트(레고 안에 레고)

**JSX**: "자바스크립트 안에 HTML, HTML 안에 자바스크립트가 가능해진 거네" — 정확한 자기 정리
- **내 질문: "JSX가 없으면 어떻게 되는데?"** → React.createElement 지옥 확인. JSX = 편의 문법
- **내 질문: "리액트도 없이 HTML/CSS/JS만 있으면 어떻게 합치냐?"** → link/script로 연결 + getElementById로 DOM 조작. 복잡해지면 지옥 → React의 존재 이유
- 최종 이해: React는 HTML과 JS를 한 곳에서 공유(JSX) + 데이터 바꾸면 화면이 자동으로(state) → "찾아서 바꾸기"를 없앰

**props**: 처음에 "뭔 말인지 모르겠어" → 붕어빵 틀 비유로 재설명 → "건네주는 건 어떻게 해?"(컴포넌트 옆에 이름="값", HTML 속성과 같은 모양) → "건네받은 건 어떻게 써?"(function(props), props.이름) → 보내는 쪽/받는 쪽 코드 쌍으로 확인
- **내 질문: "즐겨찾기 화면이 받는 쪽 아닌가?"** → 데이터를 가진 쪽이 보내는(나눠주는) 쪽. 부모=쓰는 쪽=보냄, 자식=쓰이는 쪽=받음. 사장/직원 비유.
- **내 이해: "부모가 자식한테 보내주고 자식은 카드 만들어서 다시 보내주는 느낌"** → "다시 보내는" 게 아니라 그 자리에 화면이 됨(단방향). 
- **내 질문: "화면엔 카드 컴포넌트 하나만 보이잖아"** → 틀은 하나지만 쓴 만큼 화면에 나옴(붕어빵 틀 하나로 여러 개)
- **내 결론: "그러면 정보를 for문으로 보내면 되는 거 아냐?"** — props+반복을 스스로 조합. 정확히 React의 방식(map).

**map**: **내 질문: "왜 굳이 for을 안 쓰는지"** → for는 명령(값 안 돌려줌), map은 결과 배열을 돌려줌 → JSX {}에는 값이 필요.
- **내 요약: "for은 리턴을 한 번 더 해야 하는데 map은 그냥 바로 리턴까지 해준다"** — 정확한 본질 요약.

**state**: useState 기본. 
- **내 질문 3연발: "onClick 함수는 state야?"(아님—동작과 값의 구분) / "let인데 왜 뒤에 []가 와?"(구조 분해—두 개를 순서대로 나눠 받기) / "이미 담은 건 또 못 담잖아"(if(includes)로 중복 검사—실제 앱 로직의 허점을 스스로 발견)**
- **내 질문: "그 (set)함수는 어디 있는데?"** → useState가 자동으로 만들어 줌. TV 사면 리모컨 딸려오듯.
- **내 질문: "React가 두 개를 준다는데 어디다 주는 거야?"** → `=` 왼쪽에 내가 준비한 변수에. 오른쪽 먼저 계산 → 왼쪽에 담김.
- **내 오해와 교정: "[초기값, 변한값]인가?"** → 아님. [값 하나, 그 값을 바꾸는 함수]. 값은 하나고 그게 변하는 것.
- 버튼의 `>+1<`(표면 글자)과 onClick의 `숫자+1`(계산)은 별개 — "+1이 두 번 쓰여있는데?" 질문으로 확인
- 최종: 카운터 예제 완전 이해

## Day 3 — 강의 병행 + 설계 리뷰 + 커리어 전략

### 강의 보며 나온 질문들

**Q. "className이 뭐야? 클래스 네임으로 이름 정해놓고 CSS 파일에서 불러오네"**
- 설명: className = 태그에 이름표 → CSS 파일의 `.이름표 { }`가 찾아서 꾸밈. style={{}} 직접 쓰기와 결과 같고 방식만 다름. 실전 정석은 className+CSS 분리. HTML의 class가 React에선 className(JS 예약어 충돌).

**Q. "id는 뭐야?"**
- 설명: 하나만의 고유 이름표(주민번호) vs className(반 이름, 공유 가능). CSS에서 #이름 vs .이름. React에선 id 쓸 일 적음(찾아서 조작을 안 하니까).

**내 발견: "강의하는 사람은 그냥 div로 다 감싸네. 우리는 header, section처럼 여러 코드 쓰는데"**
- 강의(div만)도 작동하지만, 우리 방식(시맨틱)이 한 단계 위. 강의 볼 때 "저 div는 우리 식으로 뭐지?" 번역하며 보기로.

**Q. "state를 쓰면 좋은 게 뭐야? state 안에 배열 쓸 수 있어?"**
- state의 가치 = 값 바꾸면 화면 자동 갱신(렌더링). 배열 state 가능 — 단 push 금지, 새 배열로 set.
- **"배열 state 이해가 안 돼"** → 숫자 state와 같은 구조로 재설명: set숫자(숫자+1) = set목록([...목록, 새것]). 둘 다 "기존 것으로 새 값 만들어 set". `...` = 상자 열어 내용물 쏟아붓기.
- **확인 문제 정답: `[...목록, "공원"]`** — 스스로 패턴 적용 성공.

**Q. "변수 두 개 써서 useState랑 똑같은 효과 낼 수 있지 않나?"**
- 실험적으로 확인: 일반 변수 2개로는 ① React한테 알리는 벨이 없어 화면이 안 바뀌고 ② 리렌더링 시 함수가 재실행되며 값이 리셋됨.
- useState = React의 금고(함수 밖 보관, 리렌더링에도 유지) + 벨(set 호출 시 리렌더링 트리거).
- **내 요약: "렌더링을 자동으로 해주고, 함수 밖에 데이터를 저장해서 꺼낼 수 있게 하고"** — 정확.

- 커밋+푸시 1건 (`8729137`)

### 내 질문과 이해 과정

**개념: `linear-gradient` (새 개념)**
- 설명: `background`는 단색뿐 아니라 그라데이션도 받는다. `linear-gradient(방향, 시작색, 끝색)`. 자리는 그대로, 넣는 값만 바뀜 — 예전 className 값 갈아끼우던 것과 같은 감각.
- **내 질문: "나 살구색 코드 몰라"** → hex 읽는 법 배움: `#RRGGBB`(빨강·초록·파랑 각 2자리). `#ff8a5a` = 빨강 꽉·초록 낮음·파랑 낮음 → 진한 코랄오렌지.
- **내 판단: "스케치 소스 기준으로 135deg로 갈게. 스케치 전부 135deg 통일이야"** → `to right`(90deg) 대신 원본과 동일한 `135deg`(왼위→오른아래 대각선)로 맞춤. 끝색도 내가 처음 고른 `#ff9a76`가 아니라 원본 값 `#ff8a5a`로 정정 — "아예 똑같이" 원칙 적용.

**개념: span 과녁 복습 (1-2)**
- 한 줄 안에서 부분만 다른 스타일 → 그 부분을 `<span className=...>`으로 떼어내 CSS 조준. (Day 2·3에서 잡은 "부분 강조는 span" 재확인)
- 정정 포인트: `🔁매년`은 알약(배경+테두리) 뱃지가 아니라 원본상 그냥 작은 흐린 글자(`font-size:11px; color:#c0aab2`)였음. 스케치 원본 확인 후 수정.

**개념: 삼항으로 className 갈아끼우기 (1-3, 이번의 핵심)**
- 고정 클래스는 `className='dday'`(따옴표), **바뀌는** 클래스는 `className={todo.done ? "todo-done" : ""}`(중괄호=JS 표현식). map이 돌 때 항목마다 done 값이 달라 서로 다른 class가 붙음.
- **내 막힘: "이해가 안 돼" → 고른 지점: "왜 p 전체가 아니라 span으로 감싸?"**
  - 해소: **CSS는 요소 하나를 통째로만 칠한다.** 한 줄 안에 체크=핑크·글자=회색 두 색이 필요한데, `<p>` 통째로 회색 주면 체크박스까지 회색이 됨 → 그래서 조각을 각자 span으로 떼어 따로 조준. (1-2의 span 과녁과 같은 원리임을 연결)
- **내 발견(맞음): "삼항연산자에 span 하나, todo.text에 span 하나 넣으면 되나"** → 정확. 체크박스 조각·글자 조각 각각 span으로 감싸는 게 맞음.
- 한 span 안에 삼항 두 개(`className={done? ...}` = 옷 고르기 / `{done? "☑":"☐"}` = 내용 고르기)가 역할이 다름을 구분.

### 남은 감각
- 색·각도는 "외우는 게 아니라 원본에서 고르는 것" — 스케치 소스가 정답지.
- 화면 일부만 다르게 = span. 상황 따라 달라지면 = className을 삼항으로.
