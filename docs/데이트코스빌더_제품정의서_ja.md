# デートコースビルダー — プロダクト定義書

> **コアフィロソフィー（このアプリのすべて）：**
> **AIは判断しない。正確なデータで候補を数多く並べ、ユーザー自身が見て選び、組み立てる。**
>
> Google Placesの実際の価格帯・営業時間・評価をそのまま表示し、ユーザーが自分で判断します。アプリが判断しないので間違えようがありません。データさえ正確に運べばよいのです。

---

## 一言定義

東京の地元住民が一日のデートコースを**自分で選んで組み立てる**アプリ。AIによるおすすめなしに、正確なデータで候補を並べ、ユーザーが選択します。

---

## コアメタファー — 「ベルトコンベア組み立て」

このアプリのアイデンティティを一言で表すなら：**ベルトコンベア。**

**今ユーザーが強いられている労働（このアプリが解消するもの）：**
デートを一つ計画するには、Googleでバラバラに検索することになります —
1. 「上野 デートスポット」検索 → タブ
2. 「その次どこ行こう」→「東京 デートスポット」検索 → 別のタブ
3. 「上野 グルメ」検索 → 別のタブ
4. 「上野 夜景」検索 → 別のタブ
5. これを全部、頭の中／メモ帳で手動で組み立てる

= **検索5回 + タブ複数 + 手動組み立て。** 毎回繰り返される労働です。

**このアプリがやること：**
その散らばった検索・組み立てを**一つの流れ（レール）につなぎ合わせます。** エリアを選ぶと候補が出て → 選ぶと次のステップ（グルメ）へ自動的に進み → 選ぶとまた次（夜景）へ → 最後にコースが完成。ユーザーが手作業でやっていた「検索→タブ→検索→タブ→組み立て」を、一本のベルトコンベアで。

**なぜこれがGoogleマップと根本的に違うのか：**
- **Googleマップ = 点を一つ探すツール**（検索）。5回別々にやらなければならない。
- **このアプリ = 点を線でつなぐツール**（組み立て）。ベルトコンベアで一度に。
- Googleマップの根本構造は「検索」であって「順次組み立て」ではありません。「デートモード」を一つ追加すれば済む話ではなく、**構造そのものが違う**のです。

**トラベルビルダーとも違う点：**
- トラベルビルダー = 空のキャンバスに自分で一つずつ検索して埋めていく
- このアプリ = **レールに沿って順番に案内**（エリア→スポット→グルメ→夜景→完成）
- デートのように「何から決めればいいか途方に暮れる」状況では、レールの方がはるかに楽です。

→ **この「ベルトコンベア」構造こそがUIフローの核心**です。一つの画面に全部詰め込まず、ステップが順番に流れるよう設計します。

## 課題（なぜ作るのか）

デートスポットを決めるのは**毎回繰り返される悩み**です。本当の原因は二つ：

1. **アイデアの枯渇** — 創造力の不足ではなく、**知っているスポットが限られている**からです。候補をたくさん見れば選べます。
2. **AIおすすめの不正確さ** — ChatGPTや既存のAIプランナーに聞くと、**価格が間違っていたり、満席の店をおすすめしたり、予約情報が間違っていたり**します。結局は自分で全部調べ直すことになります。

→ **解決策：** アプリは判断・おすすめをしません。正確なデータ（Google Places）で候補をどっさり並べ、ユーザー自身が見て判断します。

## ターゲット

**デートスポットを毎回悩む人。**（制作者本人がその証拠 = 最低ユーザー1人確保 + ニーズを100%理解）

- 旅行者向けアプリと違い、**地元住民**基準：「観光地」ではなく「自分たちの地元で行けそうな場所」
- 反復利用：デートは続くので一回きりではない
- **対象エリア = 全世界（Google Placesは国境を問わない）**：アプリは国内／海外を区別しません — 画面1.5で受け取る「エリア名」だけをGoogleに渡します。沖縄・バンコク・パリ、どこでも同じように動作します。（沖縄・タイで検証済み）
- **アイデンティティはあくまで「デートコースビルダー」（旅行アプリではない）**：全世界対応は「Googleデータなので無料でついてくるボーナス」であって、アイデンティティを「旅行アプリ」に変えるものではありません。旅行アプリ（Wanderlogなど）にアイデンティティをずらすと、差別化ポイント（日常デート特化）がぼやけます。根っこはデートコース、範囲だけ全世界。
- **利用シーン**：日常デート（関東など） + 旅行先デート（国内外）。旅行は不慣れな土地なので「何しよう」という悩みがより大きく、むしろニーズが強い。→ アプリを開く理由が増え、「利用頻度が低い」という弱点を緩和。
- **検証**：メイン利用（制作者の日常デート、関東）でまず検証 → 旅行先は構造上、自動的についてくる。拡張時に作り直すコードなし（エリアをハードコードしない）。
- **デフォルトのエリアリスト = 制限ではなく利便性**：よく行く場所をショートカットに。載せていない場所（国内外問わず）も検索・カテゴリの入り口からすべて出てきます。（沖縄でエリアリストがなくてもカテゴリですべて発見できることを検証）

## 差別化ポイント（既存アプリを使わずこれを使う理由）

競合状況をすべて確認した結果、二種類の既存アプリと明確に区別されます。

**1. 旅行プランビルダー（Wanderlog、Map Your Voyage、Travaaなど）との違い**
- 彼ら：数日間、複数都市、航空券・ホテル管理、観光客向け → 重くて複雑
- このアプリ：**一日、デート、地元**特化 → 軽くてちょうどいい

**2. AIデートアプリ（DateMore、Cupla、SoulPlanなど）との違い**
- 彼ら：AIが好みを分析しておすすめ、たまに情報が間違っている
- このアプリ：**AI判断なし。データを並べてユーザーが判断** → 間違えようがない

**一文で：** 旅行アプリではなくデートアプリであり、AIおすすめアプリではなくデータ+ユーザー判断アプリです。

## コアフロー（ユーザーがアプリで行うこと）

この検討で実際に動かしてみた、そのままの流れです：

1. **時間を決める** — その日のデートを何時から何時までやるか（例：12時〜23時）
2. **エリア別に候補を並べる** — 複数のエリアをまたぎながら、カテゴリフィルタ（見どころ・体験・カフェなど）で候補を**どっさり**表示
3. **ユーザーが選択** — 候補を見て行く場所を自分で選ぶ（アプリはおすすめせず、並べるだけ）
4. **飲食店の候補を並べる** — 選んだ場所の動線に合わせて飲食店をまたどっさり → ユーザーが選択
5. **自動組み立て** — 1の時間に合わせて、選んだものを**動線・タイムテーブル（コース）に組み立て**て表示

**重要な原則：** ユーザーに好みをインタビューしません（「雰囲気はどう？」のような質問はしない）。**カテゴリフィルタだけを渡して候補を並べます。** 選択はユーザーが行います。

## データ

- **Google Places API** — スポット・飲食店の名前、位置、営業時間、評価、価格帯
- このデータを**解釈せずそのまま表示します**（AIが「この店3千円」と言って間違える代わりに、実際の価格帯・営業時間をそのまま）
- データの壁なし：この検討で実際にGoogle Placesで候補を抽出しコースを組むことを実証しました

**データ方式：A（純粋データ、キュレーションなし）**
- アプリは「これが良いデートスポットだ」と**判断・キュレーションしません。** Google Placesの結果をそのまま並べます。
- 代わりに**フィルタをユーザーの手に握らせます** — これはキュレーションではなくツールの提供です：
  - 「評価4.0以上のみ」
  - 「今営業中のみ」
  - 「価格帯〜3千円」
  - 「カフェ / 見どころ / 体験」カテゴリタブ
- アプリは判断せずフィルタだけを渡す → 絞り込まれた結果は**ユーザー自身が作ったもの。** フィロソフィー（「ユーザーが判断」）を維持。
- **Aのトレードオフ（承知の上）：** データがGoogleマップと同じなので、差別化ポイントは①ベルトコンベア組み立て ②デート特化フィルタ、この二つにかかっています。組み立て体験が滑らかでなければ、ユーザーはGoogleマップに戻ってしまいます。だから組み立てUXが生命線です。

## MVP範囲（今作るもの）

**入れる：**
- 時間設定
- エリア別・カテゴリ別に候補を並べる（スポット + 飲食店）
- ユーザー選択
- 動線・タイムテーブルの自動組み立て
- データ：Google Places（日本語基準）

**外す（バージョン2へ）：**
- ❌ AIおすすめ
- ⚠️ ホテル — **画面・位置・評価はMVP（画面5）**、リアルタイム価格・空室・予約のみバージョン2
- ❌ イベント（花火、祭り）
- ❌ アクティビティ（特殊な体験）
- ❌ レビューの罠フィルタリング（ぼったくり・休業の自動警告）
- ❌ 多言語（英語など — 制作者が日本語でテスト可能なので後で）

**外した理由：** コア仮説（「データを並べる + ユーザーが組み立てる」が実際に使えるか）を最小限の労力で確認するのがMVPの目的です。上記の機能はそれが検証された後に乗せます。

## バージョン2以降（後で）

MVPは純粋データ（A）ですが、バージョン2の機能は**データの性格がそれぞれ異なります。** この区別が重要です：

- **イベント（花火・祭り）** — **キュレーション方式。** リアルタイムAPIがないので自分でキュレーションして入れ、**月に一度更新。** MVPの純粋データとは性格が異なる（これは人が選んだリスト）。「今月デートに良いイベント」という感じ。
- **AIおすすめ** — 判断ではなく「こういう組み合わせはどう？」程度の軽いヒントとして。MVPフィロソフィー（AI判断なし）と衝突しない範囲で。
- **ホテル予約** — ホテルの画面・位置・評価はすでにMVP（画面5）。バージョン2は**リアルタイム価格・空室・予約連携**のみ。affiliate API（楽天トラベル・Booking.comなど）。
- **アクティビティ** — 脱出ゲーム・ワンデークラスなど特殊な体験（Google Placesで一部拾えるが整理が必要）
- **レビューの罠フィルタリング** — レビューを読んでぼったくり・休業・外国人お断りを警告（重い、AI必要）
- **多言語** — 英語から → 韓国語・中国語
- **通知** — 記念日通知、予定デート通知（Webプッシュ：Service Worker・権限処理が必要 — V1には重い）
- **ダークテーマ** — 全画面のCSSを二セット管理
- **データバックアップ・復元** — V2でバックエンド（サーバー）ができれば自然に（localStorageだけでは片手落ち）
- **画像共有** — コースをきれいな画像カードにして共有（V1はテキスト+Googleマップリンク）
- **直接予約（アプリ内で完結）** — V1は公式サイトへの誘導まで
- ※ 上記の項目はV1のマイ情報タブ・共有・詳細画面を確定する際に「良いがV1範囲外」と分類されたものです（2026-07 スケッチFINAL点検にて）

**核心的な区別：**
- MVP（スポット・グルメ）= **純粋データ、キュレーションなし**（Google Placesリアルタイム）
- バージョン2イベント = **キュレーションあり、月1回手動更新**（リアルタイムデータがないため）
- この二つはデータソースも更新周期も違うので、設計時に分離して扱います。

## バージョン段階（重い順に分ける）

開発者が望むもの（一人で組み立て → 共有 → カップルアプリ）を**一度に作らず段階的に積み上げます。** カップル共有・チャットをつけると、純粋なコースビルダー（Wanderlog系）ではなくカップルアプリ（Cupla系）と競合することになり、技術も何倍も重くなるので、慎重に段階を分けます。

**MVP — 一人で使うコースビルダー（バックエンドなし）**
- マイページ（ホーム） + ベルトコンベア組み立て（画面1〜4） + カードメモ + 画像/テキスト共有
- マイページ：記念日メモ、予定コース、行ったコースの記録、お気に入り — すべてlocalStorage
- React + TypeScript + Google Places/Maps API + Vercel
- **保存ロジックは抽象化**（saveCourse/loadCourseを一箇所に）→ 後でバックエンド移行を簡単に。localStorageで始めつつ、作り直しなしで移行できるよう設計。

**バージョン2 — バックエンド + 思い出・共有（コアな付加機能）**
- **写真添付**：行ったコースにその日の写真を入れる。「デートコースに写真が自動で紐づく → アルバムを漁る必要なく『上野デート』を開けばその日の写真が全部ある。」（写真探しが面倒という悩みの解決 = キラー機能候補。写真は容量が大きいのでバックエンド/ストレージが必要）
- **重い共有**：彼女もアプリで開いて一緒に見て編集。アカウント・カップル連携。
- **彼女と写真を見ながら通話**などのカップル機能
- 記念日も共有（私たちの記念日）
- イベントキュレーション（月1回）、AIおすすめ、ホテルもこの段階
- **チャットは作らない。カカオトーク/LINEに任せる。**（すでに使っているものをまた作る必要はない）

**バージョン3 — フルカップルアプリ + コミュニティ**
- リアルタイム同期、アプリ内チャット — **本当に必要なら。**（スポットごとのコメントのような具体的ニーズがある時のみ）
- **ユーザーコースの公開・閲覧・取り込み（コミュニティ）：** ユーザーが行ったコースを（望めば）全員に公開 → 他の人が閲覧して「自分のコースに取り込む」。
  - **なぜ強力か：** 友人のフィードバック（「計画機能だけでは弱い、情報が必要」）への本当の答え。開発者が情報を自分で作るキュレーション（重い）の代わりに、**ユーザーたちのコースが互いに情報・インスピレーションになる。**「東京デート何しよう？」→ 他人のコースを閲覧 → 取り込む。制作者の元々の課題（知っていることが限られてアイデアが枯渇）をユーザーのコースが解決。
  - **なぜバージョン3か：** ユーザーが多くないとコースが溜まらず、そうしないとこの機能が輝かない（空っぽでは意味がない）。そしてバックエンド+アカウント+公開/非公開設定+コミュニティ管理（通報など）まで必要で、カップル共有（バージョン2）より重い。→ ユーザーが溜まった後。
  - 参考：トリプル(Triple)・Trip.comも「他人のプランを見て取り込む」が人気機能。ただし彼らは旅行用、これはデート用。

**コア優先順位（制作者確認済み）：** コース作成 = コア。思い出・共有 = 重要だがその次。だからコース作成から検証し、思い出・共有を乗せる順序。「最初からバックエンド」はしない — コア仮説（ベルトコンベアは楽か）を検証する前に何ヶ月もバックエンドに費やすリスク。代わりに保存ロジックの抽象化で後の移行コストをゼロに。

**友人フィードバック記録（実ユーザーの初反応）：**「トリプルの恋愛版だね。良いけど、計画を組む機能はみんなそんなに使わない。ゆくゆくはトリプルみたいに情報を教えてくれるのもあるといいな。」
- **心に留めること：** 計画機能だけでは弱い → 「計画以外にアプリを開く理由（情報）」が必要。（すでにマイノート・記念日・思い出で一部対応中、その方向を強化）
- **注意すること：** 「トリプルみたいに」を文字通り追いかけてはいけない。トリプルの情報（天気・為替・翻訳・235都市ガイド）は①旅行者向けなので地元デートには不要 ②大企業（大韓航空子会社）が数百人で作った規模 → 追いかけると重荷。開発者のアプリ流の情報 = デート特化（イベント・話題のスポット・カップル特典）、または上記「ユーザーコース公開」でユーザーに情報を作らせる。

**参考アプリ：トリプル(Triple)（旅行アプリ、大韓航空子会社）**
- **参考にすること（教本として）**：動線を見ながらプランを組むUX、コース公開・閲覧（旅行記）、カップル/同行者のリアルタイム共有、情報の添え方、ドラッグでのプラン編集。1500万ダウンロードで「動線組み+共有+情報」の組み合わせが刺さることをすでに実証 → 開発者は白紙ではなく検証済みの答案を参考にできる。詰まったら「トリプルはこれをどうやったんだ？」と開いてみる。レビューの不満（ドラッグが飛ぶなど）は先回りして避ける。
- **参考 ≠ コピー**：トリプルから**デートに合うものだけ**選んで参考にする。旅行者向け機能（航空券・ホテル予約、235都市ガイド、為替・時差・翻訳）は排除 — 地元デートには不要 + 大企業の領域。
- **トリプルと重なっても別物**：トリプル=「旅行者の旅行準備オールインワン」、このアプリ=「地元カップルの一日デート」。機能が似ていても市場・アイデンティティが違う。開発者の本当の堀 = **カップル関係中心**（彼女情報・記念日・思い出アーカイブ・繰り返される関係）— トリプルが「デートモード」を入れても旅行アプリ構造では追いつけないところ。

**遠い将来のオプション — 旅行モード（バージョン3以降）：**
- 一日のコースを**複数日に拡張** = 旅行プラン。構造上すでに可能（沖縄・タイで検証）。「コースを日付ごとに複数まとめる」。
- **ただし、デートアイデンティティは維持**：「旅行アプリ」になるのではなく「デートアプリだけど旅行もできる」ライン。トリプルとの正面衝突（航空券・ホテル予約）領域には触れない。
- ずっと後で。今やることと明確に区別する。

## 技術スタック

**MVP：**
- **フロント**：React + TypeScript（ベルトコンベア = 状態フローとして自然に表現できる）
- **データ**：Google Places API（スポット・グルメ）
- **地図**：Google Maps JavaScript API（動線表示）
- **保存**：localStorage（お気に入り・コース、バックエンドなしで）
- **デプロイ**：Vercel or Netlify（無料、GitHub連携で自動デプロイ）
- **バックエンド**：なし（フロントからAPIを直接呼び出し）

**なぜこのスタックか：**
- Webアプリから → アプリストア登録・審査・両プラットフォームの壁なしに「作ってスマホですぐ確認」。初心者にはこの速いサイクルが最も重要。
- React+TS → 開発者がすでにTypeScriptを少し知っている。このアプリはまさにReactが得意な種類（状態に応じて画面が変わる）。資料も多い。
- 後でレスポンシブにしてスマホでもアプリのように。必要になればその時ネイティブ/PWAを検討。

**バージョン2〜3で追加：**
- バックエンド（リンク共有 → アカウント・DB・リアルタイム）。イベントキュレーションデータの管理にはPythonを使える。

**注意 — コスト：**
- Google Places/Maps APIは無料枠があり、超えると課金。一人のテストは無料圏内。ユーザーが増えたら「API呼び出しキャッシング」で節約が必要（MVP段階では気にしない）。

## 画面設計（ベルトコンベアを画面に）

ベルトコンベア = 画面が順番に流れる。一つの画面に全部詰め込まずステップごとに。
**ただし、起点はマイページ（ホーム）。ベルトコンベアはホームから出発してホームに戻る循環。**

### 画面0 — マイページ（ホーム、アプリの起点）
```
┌─────────────────────────────┐
│  📌 マイノート                │
│  🎂 記念日                    │
│    彼女の誕生日 7/20 (D-16) 🔁 │ ← 日付あり、D-day自動。繰り返し有無はユーザー選択
│    私たちの100日 8/3 (D-30)  │
│  ✅ やること                  │
│    ☐ 誕生日プレゼント予約      │ ← チェックボックス、完了したら消す
│    ☐ マフラーを買う           │
│  💝 彼女情報                   │
│    • 甲殻類アレルギー          │ ← 日付なし、常時参考（完了なし）
│    • 静かなカフェが好き        │
├─────────────────────────────┤
│  📅 予定されたコース          │
│  [+ 新しいコースを作る]        │ ← 画面1へ
├─────────────────────────────┤
│  [ 行ったコース | お気に入り ] │ ← トグルボタン
│  (トグル状態に応じて一覧が変わる)│
└─────────────────────────────┘
```
- **マイノート3種（一画面に上下で、性格が異なる）：**
  - **記念日**：日付あり、D-day自動計算、**毎年繰り返すかをユーザーがチェックボックスで選択**（誕生日=繰り返し、100日=一回きり）
  - **やること**：チェックボックス、完了したら消す（todo）。日付をつけられる
  - **彼女情報**：完了なしで溜まり続ける参考リスト（アレルギー・好み）。→ **画面2・3で参考用に表示**（軽い方式、自動フィルタリングではない。ユーザーが見て判断 = フィロソフィーと一致）
- **行ったコース ↔ お気に入り = トグル**（どちらも過去の一覧で性格が似ている、スペース節約）
- **なぜホームに置くか（戦略的）**：アプリを開くと記念日・予定コース・過去の記録が見える → **デートしない日でもアプリを開く理由。**「コース作成」だけだとたまに使って忘れるが、記念日・記録があると開き続ける。→ 開発者のアプリ最大の弱点（デートは週1〜2回で利用頻度が低い）を解決。
- MVPはすべてlocalStorage。記念日共有・写真はバージョン2（バックエンド）。

**アプリ全体の流れ：**
```
画面0 (ホーム)
   ├─ [新規コース] → 画面1(時間) → 画面1.5(都/県) → 画面2(スポット) → 画面3(グルメ) → 画面4(完成)
   │                                                    画面4で[ホテル追加] → 画面5(ホテル、任意)
   │                                                    完成したら名前をつけてホームに記録
   ├─ 予定コースをクリック → コース詳細画面 (画面4を再現、閲覧モード)
   ├─ 行ったコースをクリック → コース詳細 (カードを押すとメモ・(バージョン2)写真)
   └─ お気に入りを見る
```
- **画面4 = 「コース詳細画面」として再利用**：新規作成時は編集モードで到着、保存済みコースを開く時は閲覧モードで到着。**同じ画面、モードだけ違う**（画面の再利用 = 作るものが減り一貫性が生まれる）。
- **編集モードでできること（すべて）：** スポット追加 / スポット削除（行かないと決めた場所を外す）/ 時間帯変更 / 日付変更（延期など）/ カード順序変更 / コース自体の削除（全く行かなくなった）。つまりCourseの全フィールド（itemsの追加・削除・順序、CourseItem.startTime、Course.date、coursesからの削除）を操作可能。閲覧モード = 読み取り専用。✏️（閲覧モード右上）→ 編集モード → [保存する] → 閲覧モードに復帰。
- **ホームのマイノートに[+]追加ボタン：** 記念日・やること・彼女情報の各エリアに[+] → 入力ポップアップ → 一覧に追加。各項目に✏️🗑️（共用EditDeleteButtons部品）でその場で修正・削除。やることは☐タップでdoneトグル。
- **戻る(<)ポリシー：** コース作成中は**一段階ずつ後退**（画面3→2→1.5→1、追加したスポットは維持 — 「エリアを選び直す」が可能）。画面1でもう一度戻る = ホームへ出ることなので、その時だけ**「保存せずに出ますか？」確認ポップアップ**（誤って追加したものを全部消す事故を防止）。

### 画面1 — スタート（日付・時間）
- **日付**：カレンダーからタップで選ぶ
- **時間**：開始・終了の両方、**アラーム式のホイール**で（例：開始12:00 / 終了23:00）
- エリアはここでは行わない → 次の画面
- **この入力が後の画面全体のフィルタ**：その時間に営業している場所だけが出て、その時間内にコースが配置される。だから最初の画面。

### 画面1.5 — 都/県選択（基準点の設定）
- 今回のデートの**都/県をまず決める**（東京・神奈川・埼玉・群馬・茨城・千葉…）
- **なぜ必要か（技術的）：** Google Placesは「どこ」を知らないと検索しません。県を決めるとそれが**検索の基準点**になり、以降すべての候補がその県基準でロードされます。
- **なぜ必要か（UX）：** 開発者の行動範囲が関東全域でエリアが数十個。県を先に絞ると、画面2のエリアタブに**その県のエリアだけ**が出てすっきり（ドリルダウン）。
- **80%が一つの県内デート** → 県を一度選べばその中で組む。残り20%（県をまたぐ）は「直接追加」で処理。大半のケースをシンプルに、例外は既存機能で。

### 画面2 — エリア・スポット選択（アプリの心臓）
```
💝 彼女情報 (折りたたみ/展開)：甲殻類アレルギー · 静かな場所が好き
──────────────────────────
🔍 [ スポット・エリア検索 ]        ← 何でも検索 (昭和記念公園/立川/公園)
──────────────────────────
📍 エリア: [上野][渋谷][新宿][+もっと見る]  ← 入り口1: エリアショートカット
🎯 種類: [公園][カフェ][見どころ][体験][ショッピング]  ← 入り口2: カテゴリ
価格: 安い ──●──── 高い
──────────────────────────
候補一覧 (選択に応じて変わる、県内・評価順)
  昭和記念公園 ⭐4.3 [+追加]
   ...
──────────────────────────
[+ 直接追加]  ← Googleにない場所 / 別の県の場所
   選んだ場所 3つ → [次へ: グルメ]
```

**核心構造 = 「入り口は三つ、結果は一つ」（検索窓がメイン）**
エリア・カテゴリ・検索のどれから入っても**同じ候補一覧**に集まる。入り口だけ複数、到着点は一つ → 迷わない。
**検索窓を上に大きく（メイン）、エリアタブ・カテゴリは下に補助。** 人々は検索に慣れている（Google・Instagram）ので「何か探したければ上に打つ」が直感的。

1. **まるごと検索（メイン入り口、ターゲット型A）**：検索窓に**「池袋 ネットカフェ」のようにエリア+スポットを一度に**。目的が明確な時が最速。エリアタブを経由しなくてよい。
2. **エリアで絞る（補助、インスピレーション型B）**：[上野] → 上野近くの候補。一つのエリアで**複数の場所**を見る時に良い（エリアを固定してカテゴリだけ変える）。エリアタブになければ[+]で追加（追加したエリアは記憶される→次回もタブに残る）。
3. **カテゴリで絞る（補助）**：[公園] → **県全体の公園**を評価順。← 死角の解決。
4. **組み合わせ可能**：[上野]+[カフェ] = 上野のカフェだけ。

**検索を二つに分離（穴の防止）：**「エリア追加検索」（[+]を押した時、エリアのみ）と「スポット検索」（メイン検索窓、スポットのみ）は別。同じ窓だと「池袋」をエリアとして認識すべきかスポットとして認識すべきか紛らわしい。

**ネットカフェ・カラオケなどの検索語のコツ：** MVPは日本語基準なので検索も日本語が正確（例：ネットカフェ/漫画喫茶）。開発者は日本在住なのでOK。多言語（韓国語で打っても日本語に変換して検索）はバージョン2。

**なぜカテゴリ入り口が必須か（今回発見した設計の欠陥）：**
- 問題：エリアタブだけに依存すると、制作者が**載せていないエリアが死角**になる。例）ユーザーが「東京」を選んだのに立川ボタンがないと → 昭和記念公園（立川所在）を見つけられない。「載せていないエリア = ユーザーには存在しないエリア」となり、**アプリの目的（発見の手助け）と真っ向から矛盾。**
- 解決：**カテゴリ入り口。** 立川ボタンがなくても[公園]を押せば東京全体の公園から昭和記念公園が出る。エリアリストの限界をカテゴリが埋める。
- → エリアタブ = 「よく行く場所のショートカット（利便性）」。本当の発見 = カテゴリ・検索が担当。

**検索範囲 = 県内 + 評価順：**
- 画面1.5で選んだ県内のみ（東京を選んだら東京のものだけ、他県は混ざらない）
- 評価が高いものから → 昭和記念公園のような良い場所が上に
- **県境を越えたい時（20%ケース）**：[+直接追加]で。（例：神奈川デート中に東京・お台場を入れる）

**エリアタブ = 方式C（基本リスト + 検索）：**
- 基本リスト：制作者がコードに書くエリアリスト（県別）。利便性機能であって制限ではない。
- 検索：リストになくても任意のエリア・スポットを追加可能。

**カテゴリフィルタ**：見どころ・体験・公園・ショッピングなど（カフェ・グルメは画面3）
**価格フィルタ（スライダー）**：スライダー（ルートA、Googleの4段階）。予算を超えるものは一覧から**まるごと外す**（フィルタ）。
**並び替えボタン**：評価順（デフォルト）/ 価格順。表示されているものを並べる（並び替え ≠ フィルタ）。距離順はない — ユーザーがどこから出発するか分からず基準点がないため。（画面3は追加したスポット基準があるので「徒歩N分」表示があるが、それは表示であって並び替えではない）
  - スライダー（フィルタ）で「高すぎるものは外して」+ 並び替えで「その中で安い順」→ 予算内で安いものから。両方あると強力。

**利用フロー = 二方向（Aターゲット型 / Bインスピレーション型）：**

- **方向A — 行く場所が決まっている場合（ターゲット型）**：エリアを決める → 検索窓にその場所を入力 → 候補が出る → カードで確認 → [追加]。「すでに知っているものを入れる。」
- **方向B — 決まっていない場合（インスピレーション型）**：エリアだけ決める → そのエリアの候補がずらり → **カテゴリでフィルタリングしながら流し見** → インスピレーションを得る → [追加]。「知らなかったものを発見する。」
- **Bがアプリの本当の価値**（制作者の元々の課題「知っていることが限られてアイデアが浮かばない」を解くのがB）。Aは利便性。両方が同じ画面で自然に。

**カード操作 = 中間方式：**
- カードに**基本情報はアプリ内に**表示：名前・評価・営業時間・価格・写真1枚。→ ざっと流し見する時はアプリ内で即判断。
- **詳細（レビュー・ストリートビュー・詳細位置）だけGoogleマップへ**：カードの「Googleマップで見る」→ Googleマップアプリ → 確認 → 戻って[追加]。
- なぜ中間か：方式A（全部Googleマップに渡す）は行き来が面倒で、方式B（全部アプリ内）は作るものが多い。中間 = 流し見はアプリ内、精密な確認だけGoogleマップ → 摩擦最小 + 作る負担が少ない。
- **カテゴリフィルタ**：**見どころ・体験**（カフェ・食べるものは画面3に集約 — 重複防止）
- **価格フィルタ = スライダー（ルートA）**：見た目はスライダー、裏ではGoogleの4段階（price_level）で動作。精密な円建てではない（Googleが4段階しか与えない）。精密な価格はバージョン2。
- **追加 + 選択状況**：[+追加]でカートのように集める。複数のエリアをまたいで追加するので「今何を追加したか」が常に見えなければならない。
- **[+ 直接追加]**：Google Placesにない場所（ポップアップストア・祭り・期間限定イベント・Instagramで見た新作カフェ・隠れた場所）をユーザーが直接入力してコースに入れる。
  - **名前 + 位置 + 時間を一緒に受け取る** ← 重要。位置は二つの方法：**①地図上でタップして指定 ②地名・最寄り駅で検索**（「渋谷駅」と打てばその近くに）。正確な座標でなくても「大体ここ」で十分。
  - **なぜ追加した瞬間に位置を受け取るか（設計原則）：** 名前だけ受け取って位置を空にすると、画面3の地図（ピンを刺せない）・画面4の動線・画面5のホテル近く計算がすべて壊れる。「データは作られる時に完全に埋める」→ 追加した瞬間に位置まで受け取れば、以降すべての画面がそのまま動作する。穴を入り口で塞ぐ。
  - API不要、localStorageに保存。フィロソフィーと一致（ユーザー主導）。
  - ※ **Instagramハッシュタグ連携は外す**：Metaがハッシュタグ検索APIを塞いでおり（データの壁）、Instagram投稿には構造化された場所データもない。代わりにユーザーがInstagramで見たものを「直接追加」で移して入れれば同じニーズを満たせる。
- **ここではスポットだけ**追加 → [次へ] → グルメ画面

### 画面3 — グルメ選択（地図式、カテゴリ先行入場）
```
どのスポット近く？ (タブ)
[🏮 雷門] [🌳 国立科学博物館]   ← 画面2で追加したスポットたち
──────────────────────────
🔍 [検索: "上野 オムライス"...]      ← 検索窓 (画面2を再利用)
──────────────────────────
🗺️ 地図: 選択スポット 📍 (最初はピンなしできれい)
   "📍 雷門近く · 種類を選んで"
──────────────────────────
🍜ラーメン 🍣寿司 🍖焼肉 ☕カフェ 🍰デザート 🍺居酒屋
   ↑ カテゴリを押すと → その種類だけピン+一覧を表示
──────────────────────────
[価格 ▾]                        [スキップ ›]
──────────────────────────
🍢 炉端 福笑い ⭐4.5·徒歩3分·¥¥ [+追加]
   ...ずらり (選んだ種類だけ、地図と連動)...
──────────────────────────
   追加したグルメ 1軒 → [コース完成]
```
- **位置基準が画面2と違う**：エリアタブではなく、**画面2で追加したスポットが基準。** スポットカードをタップするとそのスポットが地図の中心 → 近くのグルメがピン+一覧で。
- **★ カテゴリ先行入場（B案、ピン地獄の防止）**：最初に開くと地図はきれい + カテゴリボタンだけ。種類（ラーメン/カフェ…）を選ぶと**その種類だけ**ピン+一覧で表示。理由：近くの「グルメ全部」を最初から出すとピン数十個で地図が覆われ、かえって選べない。「並べること」と「選べるようにすること」は違う — デートで「ラーメン食べようかカフェ行こうか」のように種類から決める思考の流れとも一致。副次効果：押した時だけAPIリクエスト → Google API呼び出しの節約。
- **検索窓の追加**：「上野 オムライス」のように直接探したい時。Google Placesのテキスト検索を使用 — 画面2の検索をそのまま再利用（新規開発なし）。
- **技術実装**：カテゴリ = Google Places APIのtype/keywordパラメータで種類を指定してリクエスト → 結果をstateに → mapでピンを描く。V1で十分に実装可能。
- **なぜ地図式か**：グルメは「自分が選んだスポットの近く」が核心。一覧だけでは「どのスポットの近くか」が見えない。地図で見ると動線の感覚・距離（徒歩何分）が直感的。（画面5のホテルと同じ構造 = 再利用・一貫性）
- **地図 + 一覧の併用**：地図で「どこにあるか」、一覧で「評価・価格を比較」。ピンを押すと一覧でハイライト。
- **[スキップ]可能**：グルメはすべてのデートに必須ではない（予約済み/食べ歩き/食事の予定なし）。
- **複数追加**：候補を集める。最終配置・取捨選択は画面4で。

### 画面4 — 完成（編集可能なタイムライン）★ もう一つの心臓
- 単なる「閲覧」ではなく**編集画面。** 画面2・3で追加した候補を、画面1の時間（例：12〜23時）に合わせてタイムラインに配置。
- ユーザーが**ドラッグ&ドロップで順序を変え、捨て、時間を調整。**（「二つ以上追加して後で配置したり捨てたりする」= この画面がそれを行う）
- 追加したものが候補なので、ここで「これは入れてこれは外して」を最終決定。
- 完成したコース = 動線 + タイムテーブル。
- **カードメモ**：各カード（スポット・グルメ）をクリック → メモ入力（「3時予約済み」「出口Aで」「ここでプレゼント」）。Googleデータにはないユーザー独自の計画・文脈。localStorageにmemoフィールドで保存。（MVP）
- **共有（MVP）**：完成コースを**画像/テキストで書き出し** → カカオトーク/LINEで送る。バックエンドなしで「彼女に見せる」が可能。
- **重い共有（バージョン2）**：彼女もアプリで開いて一緒に見て編集。バックエンドが必要。
- **保存**：localStorage。**ただし、保存ロジックは一箇所に抽象化**（saveCourse/loadCourse関数）→ 後でバックエンド移行時に関数の中身だけ差し替え、アプリの残りは触らない。
- **Googleマップ経路表示**：「Googleマップで見る」ボタン → 追加したスポットが順番に打たれたコースが**Googleマップアプリで**開く（経路・交通手段・移動時間はGoogleマップが処理）。
  - MVP = **方法2（Googleマップアプリへリンク）**：開発者のアプリはリンクだけ生成、実際の経路・交通はGoogleマップが行う。初心者に易しい。ただしアプリを一時的に離れる。
  - 後で = 方法1（アプリ内地図、Google Maps JS + Directions API）：アプリ内で経路が描かれる。この検討で見たあの地図表示。余裕ができた時にアップグレード。

**→ 画面3で「複数追加」に決めたため、画面4が「編集タイムライン」になる。** もし画面3で確定選択だったら画面4は単純な閲覧だったはず。前の決定が後の画面の性格を決める例。

### 画面5 — ホテル追加（オプションのアドオン）
- 画面4で**[ホテル追加]ボタン** → この画面へ（望む人だけ）。
```
🗺️ 地図
    📍 最後のコーススポット (地図中心)
    🏨 🏨 近くのホテルたち (ピン)
──────────────────────────
🏨 ホテルA ⭐4.2 [+追加]
🏨 ホテルB ⭐4.0 [+追加]
──────────────────────────
   選ぶと → コース最後の予定として追加
```
- **なぜ画面4の次か（ホテルは位置依存）：** グルメ・見どころはコースを組む前に選んでもよいが（動線をそれらで作る）、ホテルは**デートがどこで終わるかが決まって初めて**その近くを選べる。渋谷で終われば渋谷のホテル。→ コース完成後が論理的に正しい。（画面3.5に入れるとまだ動線が未定なので「どこ近くのホテル」か分からない）
- **地図中心 + 近くのホテル方式：** 最後のスポットが地図の中心、その周辺のホテルがピンで。「歩いて行けそうか」の距離感が直感的（ホテルは「終わった場所から近いか」が核心）。Google Maps APIで実装。**画面3（グルメ）と完全に同じ地図構造 → 画面・コードの再利用、ユーザーも一度覚えれば両方同じ。**
- **データ（他の場所と異なる — 注意）：**
  - MVP：ホテルの**位置・評価・名前のみ**（Google Places）。カードを押すと予約アプリ/Googleマップへ → 実際の価格・予約はそこで。**アプリが価格を表示しないので「価格が間違っている」問題を根本から遮断**（制作者が一番痛感した課題）。
  - バージョン2：予約API（affiliate）をつないでリアルタイム価格・空室・予約まで。
  - 理由：ホテルは**価格・空室がリアルタイムで変わる。** Google Placesはこれを与えられない。カフェ・見どころ（情報が固定）と根本的に異なるデータ性格。
- **オプションのアドオン：** 基本フロー（スポット・グルメ・完成）はそのまま、ホテルは望む人だけ追加。ホテルによく行く人は押し、そうでなければ押さなくてよい。
- **ホテルを選ぶと → 画面4に復帰、コースにカードとして追加：** ホテルも他のスポット・グルメと同じタイムラインカードになる。メモをつけられ（予約情報など）、ドラッグで位置調整可能。一貫性。

**保存済みコースもいつでも再編集（完成 = ロックではない）：**
- デートの計画は変わり続ける（彼女の都合で行けない・天気・予約失敗・時間変更）。だから保存後も編集できなければならない。
- **編集 = 追加 / 削除 / ドラッグ移動：**
  - 削除：カードをスワイプまたは長押しで
  - 移動：カードをドラッグで時間帯を移す
  - 追加：**[+ スポット・グルメ追加] → 画面2へ**（検索・カテゴリで複数追加して戻る）、**[🏨 ホテル追加] → 画面5へ**
- **「追加」は種類を事前に選ばない**：[追加]を押すとただ画面2が開き、そこでグルメ・カフェ・スポット何でも複数追加して戻る。最初にコースを組む時と全く同じ方式 → 学ぶことがなく一貫している。（ホテルだけ位置依存なので画面5に例外）
- **画面の再利用**：編集用の画面を新たに作らない。既存の画面2・5を再び開くこと = 編集。作るものが減り一貫性。
- **共有との連携**：彼女が「ここ行けない」と言えば → MVPは開発者が修正して再び画像共有 / バージョン2は彼女が直接編集（重い共有）。

**参考 — ネットカフェ・カラオケ・ゲームセンターなど「滞在型」の場所は別処理不要：** 三つの入り口構造（検索・カテゴリ）のおかげで、Googleに登録された場所は検索ですべて出る。個別の場所タイプを別途面倒みる必要なし。（良い構造が「何を追加しよう」という悩みをなくす）

**データ整理（何を作って何をGoogleが与えるか）：**
- エリア基本リスト = **制作者が直接書くリスト**（唯一自分で作るデータ）
- エリア検索 = Google Places（エリア名→座標）
- スポット・グルメ候補 = Google Placesリアルタイム
- 価格・営業時間・評価 = Google Placesそのまま
→ 開発者が作るデータは実質「エリア名リスト」一つだけ。残りはGoogleが与える。

**設計原則（今回体得）：** UI（見えるもの）とデータ（裏で行われること）を常に一緒に見る。「スライダーにしよう」（UI）がGoogleデータで裏付けられるか確認 → できなければルートAのように調整。作れないものを描かない。
