### 自分の質問と理解のプロセス

**Q.「placeId ってコードを書く人が勝手に決めた名前なんじゃないの?」**
- 説明: 名前(placeId)は自分たちが決めるが、値(「ChIJ...」)は Google が世界中の場所に付けた固有番号。図書館の請求記号のたとえ — 番号は自分が作ったのではなく、図書館(Google)が付けたものを書き写しているだけ。
- 理解:「あ、理解できた。lat も同じだよね」— 名前は自分たちのもの、値は Google のものという区別を、自分から別のフィールド(lat)に応用してみた。

**Q.「コースアイテムにすでに時間があるのに、なぜ Course にもう一度 startTime/endTime が入るの?」**
- 説明: Course の時間 = デート全体の枠(12〜23時、画面1で決めたもの)。CourseItem の時間 = その中での各場所の配置(13時...)。会議のたとえ — 会議全体は9〜18時 vs 各セッションの時間。
- 理解: 同じ名前が二つの層にあるのは重複ではなく「全体 vs 個別」という層の違い。

**Q.「お気に入りの場所が理解できない」→「Place 全体にコースの場所もあるのに、お気に入りに全部混ざるんじゃないの?」**
- 説明: `type Place` は倉庫ではなく **設計図(たい焼きの型)**。実際のデータは favorites と各 course の items に **それぞれ別々に** 保存される。「すべての Place の倉庫」は存在しないので混ざりようがない。アドレス帳のたとえ — 連絡先のフォーマット(型)は同じでも、家族グループ/会社グループに別々に保存する。
- 理解: **「あ、型っていうのはたい焼きの型なんだ。それを配列で保存するってことだよね」**— 型=型、配列=入れる箱、という核心的な概念を自分でまとめた。以降ずっとこのたとえで考えるようになった。

### 学んだ設計原則
- 時間は場所(Place)ではなく配置(CourseItem)に — 同じ場所が複数のコースに違う時間で入りうるから (俳優/役どころのたとえ)
- 予定/行ってきたは保存せず日付で判断 — 状態を保存すると毎日更新が必要、日付比較なら自動
- ホテルは items の外に別扱い (画面設計とデータ構造の一致)

## Day 2 — HTML → CSS → JavaScript → React の概念を完走

### HTML

**Q.「footer とか聞いたことあるけど何? span についてもっと教えて」**
- 説明: footer/header/nav は div と機能が同じ箱だが「意味(名札)」が加わったもの。span は div と違って文字の分だけしか場所を取らない。
- 理解: 意味を持つタグの存在を知った。

**Q.「縦に積まれる、横に並ぶってどういうこと?」**
- 説明: 図で確認 — div は一行まるごと占める(賃貸まるごと一部屋)から下に積まれ、span は文字の分だけ(部屋の一角だけ)だから横に並ぶ。
- 理解: 画像を見てすぐ納得。以降「カードは div、文字の強調は span」という基準で判断するように。

**Q.「じゃあ span なしで div 一つで flex 配置すればいいんじゃないの?」**
- 説明: 技術的には半分正しい。でも ① 文章の流れの中の強調は span が自然だし ② div ばかり100個あると後でコードが読めないし ③ アクセシビリティの問題がある。
- 自分の決定: **「王道でいってくれ。人に説明するのに div だけってわけにいかないだろ」**— 発表会でコードを見せることを考えてセマンティックなタグでいくことを自分で選択。

**Q.「section って何? なんで h3 や p を使って span を使わないの? nav はボタンを追加すると自動で分けてくれるの?」**
- 説明: section=テーマのある区域。h3/p/span は「独立した見出し/段落なのか、文章中の一部なのか」で区別。nav は自動配置しない — 4等分は CSS(flex:1)がやる。
- 理解: **「タグ=意味、CSS=見た目」**が繰り返される核心だと把握。

**Q.「nav は footer で包まなくていいの?」**
- 説明: footer は位置ではなく役割(締めの情報)。アプリ下部のタブバーは純粋なメニューなので nav 単独が王道。
- 理解:「位置でタグを決めるのではなく、意味で決める」をつかんだ。

**Q.「nav の位置がどうやって自動で一番下になったの? main って何?」**
- 説明: 自動ではない — コードの順番(最後に書いたから)。画面下部への固定は CSS(position:fixed; bottom:0)。main = header/nav を除いた核心の本文。
- 理解: さっき学んだ「タグは位置を決めない」と矛盾して見えたので質問 → コードの順番 + CSS という答えで解消。前後をつなげながら学ぶパターン。

**Q.「div の箱の大きさは無限なの?」**
- 説明: 横 = 親の分だけいっぱい(制限あり)、縦 = 中身の分だけ。中身が画面を超えるとスクロール。ゴム箱のたとえ。
- 理解: 大きさが「中身と親」で決まるということ。

**Q.「じゃあなんで全部 div に入れたの?」/「nav を div の外に出してもいいんじゃない?」**
- 説明: React は一つだけ return できる → 複数のときは封筒(div)でまとめないといけない。nav を出しても結局 `<>` で包み直すことになるし、外側の div が画面全体の配置を管理するので残すほうがよい。
- 理解: **「React で一気に一つのコンポーネントとして一度に見せるってことなんだな」**— 画面一つ = コンポーネント一つ、を自分で整理。

### CSS

**復習チェック**: background(背景)/ color(色)/ padding(**「箱の中の緩衝材」**— 自分の表現)/ font-size(文字の大きさ)/ border-radius(箱を丸くする) — 翌日に5つ全部覚えていた。

**Q.(D-day の話の途中で)「日付が過ぎると D- が一つずつ減るのは HTML じゃないよね? 自動で日付を選ぶと D- が出る機能とか」**
- 説明: 正確 — HTML は死んだ文字、計算・変化は JavaScript。ウェブの3要素: HTML(骨組み)+CSS(見た目)+JS(機能)。
- 意味: まだ習っていない境界(静的 vs 動的)を自分で感知した質問。

**Q.「span なら、その span の中の小さな箱を飾るんじゃないの? どうやって自動で文字だと認識されるの?」**
- 説明: プロパティが二種類 — 文字用(color, font-size, font-weight)は文字に、箱用(background, padding, border-radius)は箱に行く。プロパティ自体が行き先を持っている。
- 理解: **「あ、CSS のコードによって違うんだ」**— プロパティごとに適用先が違うと把握。D-16 バッジ作り(color→background→padding→border-radius と積み上げる)で確認。

### flexbox (深掘り)
- justify-content = 主軸(横)の整列、align-items = 交差軸(縦)の整列 — 「理解できた、そういう意味なんだ」
- flex-direction: row/column、gap、flex:1(スペースを均等に分配 — タブバー4等分の秘密)
- 以前の質問(「nav は自動で4等分される?」)の答えが flex:1 だったことを確認

### JavaScript 6つのレンガ

1. **変数**: let(変わる)/const(固定)—「うちのデートは const、残りの日数は let」正解
2. **値の種類**: string/number/boolean —「文字/数字/真偽」正解
3. **計算・連結**: `"入れた場所 " + 4 + "件"` — 正解。空白は引用符の中に書いたそのまま出る、という細かい点も追加。
4. **判断(if)**: rating >= 4.0 →「いいね」正解。`===`(比較)vs `=`(代入)の区別
   - **自分のつなげる質問:「これを学ぶと、じゃあデートの日付が過ぎたら自動で履歴に移るの?」**— if を学んだ途端にアプリの動作(予定/行ってきたの自動分類)と結びつけた。「移すのではなく、見るたびに判断する」という構造を理解。
5. **繰り返し(for)**: for...of。「プレゼント予約/マフラーを買う/ケーキ注文」3回出力、正解
   - **自分の質問:「繰り返しをもう少し具体的に」**→ スローモーション(一周ごとに箱の値が上書きされる)+ 繰り返しの中で計算・判断を組み合わせて実行するのを確認
   - **自分の発見:「Java の for 文とちょっと違う見た目だね」**→ Java の拡張 for(`:`)と JS の for-of(`of`)は同じ概念で、記号だけが違う
6. **関数**: function 名前(引数){return} — Java のメソッドと同じ(型は省略)
   - **自分の質問:「引数は外に作っておいてから始めるんだよね?」**→ 違う。引数=関数の括弧の中の空の箱(宅配ボックス)、実引数=呼ぶときに入れる値(荷物)。外の変数でも直接の値でも構わない。
   - **自分の質問:「引数が必要な関数だと、変数にどうやって入れるの?」**→ 入れるときは実引数なしで関数そのものだけ(機械を買っておく)、呼ぶときに実引数(材料を入れて回す)
   - **自分の発見:「JavaScript は関数そのものを変数に設定できるんだ」**— Java との本当の違いを把握。React の onClick への伏線。

### React 5つの核心

**コンポーネント**: 画面を return する関数。「コンポーネントは一つの大きな関数の中に複数の構造が混ざったもの?」→ 違う、**小さく分けた関数たちの組み立て**。一つのコンポーネント = 一つの役割。
- 最終的な理解: **「関数(=コンポーネント)の中に複数のコンポーネントが合わさったものを画面に見せるってことだよね」**— コンポーネントの中にコンポーネント(レゴの中にレゴ)

**JSX**:「JavaScript の中に HTML、HTML の中に JavaScript ができるようになったってことだね」— 的確な自己整理
- **自分の質問:「JSX がなかったらどうなるの?」**→ React.createElement 地獄を確認。JSX = 便利な文法
- **自分の質問:「React もなしで HTML/CSS/JS だけだと、どうやって合わせるの?」**→ link/script でつなぐ + getElementById で DOM を操作。複雑になると地獄 → React が存在する理由
- 最終的な理解: React は HTML と JS を一箇所で共有(JSX)+ データを変えると画面が自動で(state)→「探して書き換える」をなくす

**props**: 最初は「何の話かわからない」→ たい焼きの型のたとえで再説明 →「渡すのはどうやるの?」(コンポーネントの横に 名前="値"、HTML の属性と同じ形)→「渡されたのはどう使うの?」(function(props)、props.名前)→ 送る側/受け取る側のコードのペアで確認
- **自分の質問:「お気に入り画面が受け取る側なんじゃないの?」**→ データを持っている側が送る(分け与える)側。親=使う側=送る、子=使われる側=受け取る。社長/社員のたとえ。
- **自分の理解:「親が子に送って、子はカードを作ってまた送り返す感じ」**→「送り返す」のではなく、その場で画面になる(一方向)。
- **自分の質問:「画面にはカードコンポーネント一つしか見えないじゃん」**→ 型は一つだが使った分だけ画面に出る(たい焼きの型一つで複数個)
- **自分の結論:「じゃあ情報を for 文で送ればいいんじゃないの?」**— props+繰り返しを自分で組み合わせた。まさに React のやり方(map)。

**map**: **自分の質問:「なんでわざわざ for を使わないのか」**→ for は命令(値を返さない)、map は結果の配列を返す → JSX の {} には値が必要。
- **自分のまとめ:「for はリターンをもう一回しないといけないけど、map はそのまま直接リターンまでしてくれる」**— 的確な本質のまとめ。

**state**: useState の基本。
- **自分の質問3連発:「onClick 関数は state?」(違う—動作と値の区別)/「let なのになんで後ろに [] が来るの?」(分割代入—二つを順番に分けて受け取る)/「すでに入れたものはまた入れられないじゃん」(if(includes)で重複チェック—実際のアプリロジックの穴を自分で発見)**
- **自分の質問:「その (set)関数はどこにあるの?」**→ useState が自動で作ってくれる。テレビを買うとリモコンが付いてくるように。
- **自分の質問:「React が二つくれるって、どこにくれるの?」**→ `=` の左に自分が用意した変数に。右を先に計算 → 左に入る。
- **自分の誤解と修正:「[初期値, 変わった値] なの?」**→ 違う。[値一つ, その値を変える関数]。値は一つで、それが変わる。
- ボタンの `>+1<`(表面の文字)と onClick の `数字+1`(計算)は別物 —「+1 が二回書いてあるけど?」という質問で確認
- 最終: カウンターの例を完全に理解

## Day 3 — 講義の並行 + 設計レビュー

### 講義を見ながら出た質問たち

**Q.「className って何? クラスネームで名前を決めておいて CSS ファイルから呼び出してるね」**
- 説明: className = タグに名札 → CSS ファイルの `.名札 { }` が探して飾る。style={{}} で直接書くのと結果は同じで、やり方だけ違う。実戦の王道は className+CSS の分離。HTML の class が React では className(JS の予約語との衝突)。

**Q.「id って何?」**
- 説明: 一つだけの固有の名札(マイナンバー)vs className(クラス名、共有可能)。CSS で #名前 vs .名前。React では id を使うことは少ない(探して操作をしないから)。

**自分の発見:「講義してる人はただ div で全部包んでるね。うちは header, section みたいに色々なコードを使うのに」**
- 講義(div だけ)でも動くが、うちのやり方(セマンティック)が一段上。講義を見るときは「あの div はうちの流儀だと何?」と翻訳しながら見ることに。

**Q.「state を使うと何がいいの? state の中に配列を使える?」**
- state の価値 = 値を変えると画面が自動で更新(レンダリング)。配列 state も可能 — ただし push 禁止、新しい配列で set。
- **「配列の state が理解できない」**→ 数字の state と同じ構造で再説明: set数字(数字+1) = set一覧([...一覧, 新しいもの])。どちらも「既存のもので新しい値を作って set」。`...` = 箱を開けて中身をぶちまける。
- **確認問題の正解: `[...一覧, "公園"]`**— 自分でパターンを適用できた。

**Q.「変数を二つ使えば useState と同じ効果を出せるんじゃないの?」**
- 実験的に確認: 普通の変数2つでは ① React に知らせるベルがないので画面が変わらず ② 再レンダリング時に関数が再実行されて値がリセットされる。
- useState = React の金庫(関数の外に保管、再レンダリングでも維持)+ ベル(set 呼び出し時に再レンダリングをトリガー)。
- **自分のまとめ:「レンダリングを自動でしてくれて、関数の外にデータを保存して取り出せるようにしてくれる」**— 的確。

### 自分の質問と理解のプロセス

**概念: `linear-gradient` (新しい概念)**
- 説明: `background` は単色だけでなくグラデーションも受け取る。`linear-gradient(方向, 開始色, 終了色)`。場所はそのままで、入れる値だけが変わる — 以前 className の値を差し替えていたのと同じ感覚。
- **自分の質問:「杏色のコードを知らない」**→ hex の読み方を学ぶ: `#RRGGBB`(赤・緑・青が各2桁)。`#ff8a5a` = 赤いっぱい・緑低め・青低め → 濃いコーラルオレンジ。
- **自分の判断:「スケッチのソースを基準に 135deg でいく。スケッチは全部 135deg で統一だ」**→ `to right`(90deg)ではなく元と同じ `135deg`(左上→右下の対角線)に合わせた。終了色も自分が最初に選んだ `#ff9a76` ではなく元の値 `#ff8a5a` に訂正 —「まったく同じに」の原則を適用。

**概念: span で狙う復習 (1-2)**
- 一行の中で一部だけ違うスタイル → その部分を `<span className=...>` で切り出して CSS を狙う。(Day 2・3 でつかんだ「一部の強調は span」を再確認)
- 訂正ポイント: `🔁毎年` は錠剤(背景+枠線)のバッジではなく、元では単なる小さい薄い文字(`font-size:11px; color:#c0aab2`)だった。スケッチ原本を確認して修正。

**概念: 三項で className を差し替え (1-3、今回の核心)**
- 固定のクラスは `className='dday'`(引用符)、**変わる** クラスは `className={todo.done ? "todo-done" : ""}`(波括弧=JS の式)。map が回るとき項目ごとに done の値が違うので、それぞれ別の class が付く。
- **自分のつまずき:「理解できない」→ 引っかかった箇所:「なんで p 全体じゃなくて span で包むの?」**
  - 解消: **CSS は要素一つをまるごとしか塗れない。** 一行の中にチェック=ピンク・文字=グレーという二色が必要なのに、`<p>` まるごとにグレーを与えるとチェックボックスまでグレーになる → だから各パーツをそれぞれ span で切り離して個別に狙う。(1-2 の「span で狙う」と同じ原理だと結びつけた)
- **自分の発見(正しい):「三項演算子に span 一つ、todo.text に span 一つ入れればいいのかな」**→ 正確。チェックボックスのパーツ・文字のパーツをそれぞれ span で包むのが正しい。
- 一つの span の中に三項が二つ(`className={done? ...}` = 服を選ぶ /`{done? "☑":"☐"}` = 中身を選ぶ)で、役割が違うことを区別。

### 残った感覚
- 色・角度は「覚えるものではなく、元から選ぶもの」— スケッチのソースが答案。
- 画面の一部だけ違う = span。状況によって変わる = className を三項で。

## Day 5 — Phase 1-4〜1-6 (flex のネスト実戦、一日で3つ)

> 基準: `docs/sketches_final/screen0_home_FINAL.png`。3つの作業は結局 **「flex の親をどこに置くか」** という一つの問題だった。

### やったこと
- **1-4. タブバー2層**: アイコン/文字を span で分けて `flex-direction: column` + `gap:4px`。ホームだけ `active` ピンク
- **1-5. 領域ごとの ＋ ボタン**: h3 に `display:flex` + `justify-content: space-between` (タイトルは左 · ＋ は右)
- **1-6. コースカード**: 🌸 / 内容 div / › を直系の子3つに。`align-items:center` + 内容に `flex:1`
- ダミーデータに `startTime`/`endTime` を追加 (データ設計図の `Course` 型に合わせる)、`region` を削除
- コミット+プッシュ 2件 (`57cc2db`, `9504bb6`)

### 自分の質問と理解のプロセス

**自分の発見(重要):「ホームだけピンクなのは、ホーム画面だからじゃないの? 後でホーム画面じゃないときはピンクじゃないでしょ」**
- ハードコードするよう指示を受けて **その値が仮のものだと自分で見抜いた。** ピンクは「ホームボタンの色」ではなく「今見ている画面のタブの色」だ。
- だから名前が `home-pink` ではなく `active` であるべき — 後でどのタブに付いても意味が通る名前。今ホームに埋めた `active` は **ダミーデータと同じ性格**(見た目確認用の仮の値)。
- 理解: 名前をうまく付けておけば、1-7・ルーティング以降 **CSS は一文字も直さず** に JSX に埋めた値だけ三項で変えれば終わる。(Day 4 の「三項で className を差し替え」がここにつながる)

**Q.「jsx を分けるって何?」**
- 説明: `<button>🏠ホーム</button>` では 🏠 とホームは **文字の一かたまり** なので個別に狙えない。`font-size:24px` を与えると両方 24px になる。
- 結びつけ: **Day 4 の D-16 で span を狙ったのと同じ状況。** あのときは一つのパーツ(D-16)だけ狙えばよく、今回は両方のパーツが必要というだけ。
- 理解: 名前は `tab-icon` /`tab-label` にした —「一週間後の自分が分かる名前」(span1/span2 禁止)。

**自分の質問(自力で答えを当てた):「gap を入れると、ボタン同士じゃなくてボタンの中にある子要素に行くのかな」**
- 正確。**`gap` はそのルールが狙った要素の「直系の子どうしの間」に入る。**
  - `.tab-bar` に gap → 子は button 4個 → ボタン同士が広がる
  - `.tab-bar button` に gap → 子は span 2個 → アイコン↔文字の間が広がる
- **「直系」が核心 — gap は孫には流れない。** 各 flex 親は自分の子だけを担当する。
- 派生の理解: **button の二重の身分** — 上から見れば `.tab-bar` の子、下から見れば span 2個の親。これが flex ネストのすべて。
- (ボタン同士の間隔は gap ではなく `flex:1` が処理中 — 4個が余った幅を均等に分け合って、それぞれ自分の枠の中で中央に立つから。Day 2 の「タブバー4等分の秘密」がここで再確認された)

**概念: 優先順位(specificity) (新しい概念)**
- `.tab-bar button` にすでに `color:#8a8a99`(グレー)があるのに `.active` でピンクを与えると **どっちが勝つ?**
- ルール: **クラスの数を先に数え、同じならタグの数を数える。**
  - `.tab-bar button` = クラス1 + タグ1
  - `.active` = クラス1 + タグ0 → クラス 1:1 で引き分け → タグ 0:1 で **負ける** (グレーのまま)
  - `.tab-bar .active` = クラス2 → 最初の勝負で **勝つ** ✅
- 二つ目の理由(スコープ): `.tab-icon` のように接頭辞なしで置くと、他の画面が同じ名前を使うとき干渉する。**部品の CSS はその部品の中だけで振る舞う** → `.tab-bar .tab-icon` に訂正。
- 適用: 1-5 では言われていないのに `.my-note .plus-button` と自分で接頭辞を付けた。

**概念: 主軸 vs 交差軸 (今日いちばん重要)**
- `flex-direction` を変えると `justify-content` と `align-items` の **意味も一緒に反転する。**

| flex-direction | 主軸 | justify-content | align-items |
|---|---|---|---|
| `row`(デフォルト) | 横 → | 左右 | 上下 |
| `column` | 縦 ↓ | 上下 | 左右 |

- **覚えるのは方向ではなくルール一つ:** justify はいつも主軸、align はいつも交差軸。主軸がどこかは `flex-direction` が決める。
- 今日は実際に両方使った — タブバーの button は `column`(主軸=縦)、コースカードは `row`(主軸=横)。方向で覚えていたら必ず間違えていた場面。
- (Day 2 で「justify=横、align=縦」と理解していたことの **訂正・深化**。あれは `row` のときだけ正しかった)

**自分のつまずき:「構造の部分がよく分からない」(1-6、今日の最大の難関)**
- 1-4・1-5 は flex 一つだったが、1-6 は **flex の中に flex** + `flex:1` + `align-items` が一度に出てきた。しかも「画面に見えない div をなぜ作るのか」が初めてだった。
- 段階ごとに画面がどう見えるかで解消:
  1. flex を付ける前 → h3/p が block なので勝手に縦に積まれる (ブラウザのデフォルト動作)
  2. `display:flex` だけ付けると → **タイトルとサブタイトルが横に並んで** 立ってしまう (flex が子の元々の性質を上書きする)
  3. 🌸・› をそのまま追加すると → 子が4つ → `🌸 タイトル サブ ›` 全部が横
- **つまずく箇所:** 「タイトルとサブタイトルだけ縦にして」と親に頼む方法がない。flex の親には **「子全部横」か「子全部縦」** かしかなく、一部だけ違うのは不可能。
- **発想の転換:** ❌「親にこの二つだけ違うようにして」→ ⭕ **「この二つを一つにまとめて、親に子一つとして見せよう」**
- **まとめる div の正体:** 画面に何も描かないのになぜあるかというと — **flex の射程を断つため。** div で包んだ瞬間、カードの「横に!」という命令が div で止まり、その中では元の性質(block=縦)が復活する。(たとえ: div は扉を閉じた部屋。カードは「部屋一つ」を一列に並べるだけで、部屋の中までは命令が届かない)

**概念: `flex: 1` — 余った幅を食べる (`space-between` と何が違うか)**
- flex の子はデフォルトで **自分の中身の分だけ** 幅を取る → 三つが左にぎゅっと寄って右がガラ空きに。
- 選択肢A `space-between`: 余った空間を **子どうしの間にまく。** 1-5 は子が **二つ** なので間が一つだけ → まるごと真ん中に寄って二つが両端に押される(きれい)。でも 1-6 は子が三つなので **タイトルが真ん中に浮いてしまう**(スケッチと違う)。
- 選択肢B `flex:1`(採用): 真ん中の div を **太らせて** › を右端へ **押し出す。** 🌸 は左にぴったり、タイトルはそのすぐ隣を維持。
- **一行の違い:** `space-between` は **空きを間に入れる** こと、`flex:1` は **一人を太らせて押し出す** こと。結果が似て見えても原理は正反対。
- **同じ `flex:1` なのに結果が違った理由:**

| | 誰が `flex:1`? | 結果 |
|---|---|---|
| タブバー | ボタン4個 **全部** | 余った幅を 1:1:1:1 で **分け** 合う → 均等 |
| コースカード | div **一つだけ** | 余った幅を **一人で全部** 食べる → 隣を押し出す |

- ルールは一つ: `flex:1` = 「余った空間の分配に参加、持分1」。参加者が四つなら 1/4 ずつ、一つなら独り占め。🌸・› は `flex:1` がないので **分配に不参加** → 自分の大きさのまま。
- **同じプロパティなのに「誰が持っているか」で結果が変わる** — 理解した/覚えた、の分かれ道。

**概念: `align-items: center` が必要だった理由**
- ないと 🌸 と › が **上に張り付く。** flex の子のデフォルトが `stretch`(交差軸へ親の高さの分だけ伸びる)なので、🌸/› の span がカードの高さの分だけ伸びた **透明な箱** になり、文字はその箱の一番上に座る。
- `align-items:center` = 「伸ばさずに交差軸の中央に置け」。

**概念: 生のテキストも flex の子である**
- `<h3>🎂 記念日 <button>+</button></h3>` — span で包んでいないのに `space-between` が効いた。flex が生のテキストを **見えない箱に入れて** 子として扱うから(匿名 flex アイテム)。
- ただし、**そのテキストだけを個別に狙う的はない** → 後でタイトルの色だけ変えたくなったら、そのとき包む必要がある。今は包まなくても動作は同じ。

### 今日引っかかったこと (ミスノート)

**1. 保存していないのに CSS を疑った**
- `flex-direction: column` を入れたのに画面が「そのまま」。CSS の問題だと思ったが **JSX の未保存** が原因。チャットには貼り付けたがファイルには入っていなかった。
- **確認法(今後の武器):** dev サーバーのログに `hmr update /パス` が出たかを見る。CSS は二回出ていたが JSX は一度も出ていなかった → 出ていない = 変わっていない。
- なぜ「そのまま」だったか: CSS はちゃんと入ったが button の中が相変わらず `🏠ホーム` の一かたまり → **積む材料がなかった** だけ。

**2. undefined は静かに消える**
- データから `region` を抜いたのに JSX に `📍{course.region}` が残った。React は undefined を何も描かないので **エラーなく 📍 だけ** ぽつんと残った。
- 一番見落としやすい種類 — 落ちずに静かにおかしくなる。

**3. 区切り点は間だけに**
- `… · 4件 ·` のように最後にも `·` が付いた。**`gap` が子の「間だけ」に入るのと同じ感覚。**

**4. Claude もファイルを見ないと間違える**
- 「h3/p のデフォルト margin でカードが浮くはず」という指摘を受けたが **浮かなかった。** `index.css` に `* { margin: 0 }` のリセットがすでにあったから。
- 自分がすでに CSS リセットをしておいたおかげで、`.my-note h3` の margin も「デフォルト値の上書き」ではなく **「0 から欲しいだけ与える」** だった。はるかに予測しやすいやり方。
- **教訓: ファイルを見ずに一般論で決めつけると間違える — 自分にも AI にも同じく当てはまる。** AI の言うことも一度は疑うこと。

### データ設計図の遵守 (1-6)

- `startTime:"12:00"`, `endTime:"23:00"` の追加は画面に時間を出すためではなく **設計図の `Course` 型が元々そういう形だから。**

- **知らずに置いたものと、分かって置いたものはまったく違う。**
